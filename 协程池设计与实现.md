**实现一个并发任务处理器：给定一个包含100个任务ID的列表，要求控制最大并发数为3，模拟并发调用某个外部接口（如打印ID）**

**需求 1：处理 100 个任务。** -> 这意味着要写个 `for` 循环。

**需求 2：并发调用外部接口。** -> 这意味着要在 `for` 循环里使用 `go func()` 开启协程。

**需求 3（核心矛盾）：最大并发数为 3。** -> 如果只用 `for` + `go func()`，瞬间就会启动 100 个协程，这不符合要求。必须有一个“卡脖子”的机制。

**隐形需求：主程序不能提前退出。** -> 必须等这 100 个任务全部跑完，程序才能结束。



**怎么“卡脖子”，保证最多只有 3 个在跑？** * *思考：* 什么东西能设定固定容量，满了就阻塞住？

- *敲定：* **带缓冲的 Channel**。设置容量为 3，把它当成一个“令牌桶”。只有拿到令牌的人才能干活，干完活把令牌还回来。

**怎么“等大家全部干完活”再结束？**

- *思考：* 需要一个计数器，派发一个任务加一，干完一个减一，减到 0 就放行。
- *敲定：* **`sync.WaitGroup`**。



带缓冲的 Channel (Semaphore 信号量机制) + sync.WaitGroup:

```go
package main
import(
    "fmt"
    "sync"
    "time"
)
//模拟外部接口调用
func processTask (id int){
    fmt.printf("Goroutine正在处理任务 ID: %d\n",id)
    time.Sleep(100*time.Millisecond) // 模拟网络耗时
}

func main(){
    //准备100个任务id
    taskIds:=make([]int,100)
    for i:=0;i<100;i++ {
		taskIds[i]=i+1
    }
    
    maxlimit:=3
    // 核心：利用带缓冲的 channel 作为信号量控制并发
    // 使用 struct{} 是为了做到零内存分配
    limitch:=make(chan struct{},maxlimit)
    
    var wg sync.WaitGroup
    
    //遍历任务，派发 Goroutine
    for _,id:=range taskIds{
        wg.Add(1)
        
        // 向 channel 写入数据，相当于获取一个“并发令牌”
        // 如果 channel 已满（达到 3 个），这里会阻塞，直到有 Goroutine 释放令牌
        limitch<-struct{}{}
        
        //开启协程处理任务
        go func(taskID int){
            // 确保无论是正常结束还是 panic，都能正常释放令牌和 wg
            defer wg.Done()
            defer func(){<-limitch}()// 从 channel 读取数据，释放“令牌”
            processTask(taskID)
        }(id)
    }
    //阻塞等待所有任务执行完毕
    wg.Wait()
}
```

 **灵魂拷问一：为什么要用 `chan struct{}` 作为信号量，而不是 `chan int` 或 `chan bool`？**

- **满分回答：** “因为空结构体 `struct{}` 在 Go 语言中**不占用任何内存空间（大小为 0）**。我们使用这个 channel 仅仅是为了它的‘阻塞’和‘唤醒’特性（作为信号量），不需要真正传递数据。使用 `chan struct{}` 可以将内存开销降到极低，这是一种非常地道的 Go 性能优化写法。”

**灵魂拷问二：在启动 goroutine 时，为什么要在匿名函数末尾加上 `(id)`，直接在内部用外层的 `id` 不行吗？**

- **满分回答：** “这是一个 Go 语言中非常经典的**闭包变量捕获（循环变量捕获）陷阱**。在 Go 1.22 版本之前，`for` 循环中的变量 `id` 是共用同一块内存地址的。如果直接在闭包里使用外层的 `id`，由于 `for` 循环的执行速度远快于 Goroutine 的启动速度，会导致大部分 Goroutine 最终打印出来的都是最后一个任务的 ID（即 100）。虽然 Go 1.22 修复了这个历史遗留问题，但为了代码的严谨性和向下兼容，显式地将变量作为参数传递给闭包依然是最佳实践。”

### 协程池设计与实现

协程池是一种管理goroutine生命周期和数量的技术，类似于线程池。

它通过**限制同时运行的goroutine数量**，达到资源合理分配和避免资源耗尽的目的。

协程池通常包含任务队列和固定数量的worker goroutines两部分，任务队列用于存放待处理的任务，worker goroutines负责从队列中取出并执行任务。

```go
package main
import (
	"fmt"
	"sync"
	"time"
)
// GoroutinePool 结构体定义
type GoroutinePool struct{
    taskChan chan func()
    wg		 sync.WaitGroup
}
// NewGoroutinePool 创建协程池
func NewGoroutinePool(size int)*GoroutinePool{
    pool:= &GoroutinePool{
        taskChan: make(chan func(), size),
    }
    pool.wg.Add(size)
    for i := 0; i < size; i++ {
		go pool.worker()
	}
	return pool
}
// worker 工作协程逻辑
func (p *GoroutinePool) worker() {
	defer p.wg.Done()
	for task := range p.taskChan {
		task()
	}
}
// Submit 提交任务
func (p *GoroutinePool) Submit(task func()) {
	p.taskChan <- task
}
// Shutdown 关闭协程池
func (p *GoroutinePool) Shutdown() {
	close(p.taskChan)
	p.wg.Wait()
}

func main(){
    pool := NewGoroutinePool(10)
    for i := 0; i < 40; i++ {
		i := i
		pool.Submit(func() {
			time.Sleep(time.Second)
			fmt.Printf("Task %d done by %d\n", i, runtime.NumGoroutine())
		})
	}

	time.Sleep(5 * time.Second)
	pool.Shutdown()
}
```

### 限流器

**使用滑动窗口，从100个ip地址里面过滤10个出来**

模拟瞬间收到 100 次 IP 请求，利用滑动窗口算法（设定窗口大小例如 1 秒，最大阈值为 10），最终成功放行 10 个请求，拒绝 90 个请求。

既然是滑动窗口限流，它的底层逻辑其实就是一个**时间戳队列**：

1. **数据结构选型：** 需要一个队列（在 Go 里可以用切片 `[]time.Time`）来记录每次请求到达的时间戳。

2. **并发安全控制：** 100 个请求可能是并发打过来的，所以操作这个切片必须加锁（`sync.Mutex`）。
3. **窗口滑动的核心逻辑：**
	- 每次新请求到来时，获取当前时间 `now`。
	- 检查队列里现有的时间戳，**把那些早于 `now - window_size`（已经掉出窗口外）的时间戳全部踢掉**（这就是“滑动”的动作）。
	- 检查清理后的队列长度：如果 `< 10`，说明还有额度，把当前时间戳塞进队列，返回 `true`（放行）；如果 `>= 10`，说明额度满了，直接返回 `false`（拦截）。

```go
package main

import(
	"fmt"
    "sync"
    "time"
)
// SlidingWindowLimiter 定义滑动窗口限流器
type SlidingWindowLimiter struct {
	mu           sync.Mutex      // 互斥锁，保证高并发下的数据一致性
	timestamps   []time.Time     // 记录请求时间戳的队列
	windowSize   time.Duration   // 窗口大小 (例如 1 秒)
	maxAllowed   int             // 窗口内最大允许通过的请求数 (题意：10个)
}
// NewSlidingWindowLimiter 初始化限流器
func NewSlidingWindowLimiter(windowSize time.Duration, maxAllowed int) *SlidingWindowLimiter {
	return &SlidingWindowLimiter{
		timestamps: make([]time.Time, 0),
		windowSize: windowSize,
		maxAllowed: maxAllowed,
	}
}
// Allow 判断当前请求是否允许通过 (核心逻辑)
func (l *SlidingWindowLimiter) Allow() bool {
    l.mu.lock()
    defer l.mu.Unlock()
    now:=time.Now()
    // 计算窗口的起始边界
	windowStart := now.Add(-l.windowSize)
    // 1. 滑动窗口：清理掉已经过期（掉出窗口）的时间戳
	// 找到第一个在窗口内的时间戳索引
    validIndex := 0
    for i, t := range l.timestamps {
		if t.After(windowStart) {
			validIndex = i
			break
		}
		// 如果全都在窗口外，validIndex 最终其实需要把整个切片清空
		if i == len(l.timestamps)-1 {
			validIndex = len(l.timestamps)
		}
	}
    // 截取切片，只保留有效窗口内的记录 (这里是 Go 切片操作的精髓)
	l.timestamps = l.timestamps[validIndex:]
    // 2. 判断是否超过限流阈值
	if len(l.timestamps) < l.maxAllowed {
		// 未超限，记录当前时间戳，予以放行
		l.timestamps = append(l.timestamps, now)
		return true
	}

	// 超过阈值，直接拦截
	return false
}
func main() {
	// 初始化一个限流器：窗口为 1 秒，最多允许 10 个请求通过
	limiter := NewSlidingWindowLimiter(1*time.Second, 10)

	// 模拟 100 个 IP 并发请求
	totalRequests := 100
	var wg sync.WaitGroup

	// 统计结果
	var allowedCount, rejectedCount int
	var countMu sync.Mutex

	fmt.Println("开始模拟 100 个 IP 并发请求打入网关...")

	for i := 1; i <= totalRequests; i++ {
		wg.Add(1)
		go func(ipID int) {
			defer wg.Done()
			
			// 调用滑动窗口进行判断
			if limiter.Allow() {
				// 放行
				countMu.Lock()
				allowedCount++
				countMu.Unlock()
				fmt.Printf("[放行] IP-%03d 成功获取服务\n", ipID)
			} else {
				// 拦截
				countMu.Lock()
				rejectedCount++
				countMu.Unlock()
				// 为了日志整洁，拦截的就不全打印了
			}
		}(i)
	}

	wg.Wait()
	fmt.Println("--------------------------------------------------")
	fmt.Printf("处理完毕！共放行（过滤出）: %d 个 IP 请求，拦截: %d 个请求\n", allowedCount, rejectedCount)
}
```

### 设计模式

GoF提出的设计模式有23个，包括：

（1）创建型(Creational)模式：如何创建对象；

（2）结构型(Structural )模式：如何实现类或对象的组合；

（3）行为型(Behavioral)模式：类或对象怎样交互以及怎样分配职责。

#### 1.创建型模式（Creational）—— 解决“怎么优雅地 new 对象”

- **单例模式 (Singleton)**：是保证一个类仅有一个实例，并提供一个访问它的全局访问点。
	- **核心思想**：全天下只能有这一个实例。
	- **实战**：你刚才手撕的 `GetGlobalFactory()`（配合 `sync.Once`），以及全局的数据库连接池 `Redis.Client`。
- **工厂模式 (Factory)**：
	- **核心思想**：你不要自己去 `new` 复杂的对象，你告诉工厂你要什么，工厂给你造好。
	- **实战**：你的 `AIModelFactory`。把 `switch-case` 变成 `map` 路由表，消灭了业务层创建对象的脏活累活。

#### 2. 结构型模式（Structural）—— 解决“不同接口怎么拼装拼接”

- **适配器模式 (Adapter)**：
	- **核心思想**：Type-C 转 HDMI 扩展坞。把原本不兼容的第三方接口，翻译成你的系统能听懂的统一标准。
	- **实战**：你的 `mcpToolAdapter`，把远程的 MCP JSON-RPC 协议，完美包装成了 Eino 框架认识的 `tool.InvokableTool`。
- **装饰器模式 (Decorator)**：
	- **核心思想**：在不改变原有函数功能的前提下，给它“穿上一层又一层衣服”来增加新功能。
	- **实战**：Gin 框架里的 **Middleware（中间件）**！比如你写了一个计算耗时的中间件，或者校验 JWT 的中间件，把它套在原有的 API 路由外面，这就是典型的装饰器。

#### 3. 行为型模式（Behavioral）—— 解决“对象之间怎么发消息、定规矩”

- **策略模式 (Strategy)**：
	- **核心思想**：完成同一件事有很多种算法，把它们各自封装起来，并且可以相互替换。
	- **实战**：这在 Go 语言中用 **Interface（接口）** 实现得最多。比如支付场景，有 `微信支付`、`支付宝支付`。你定义一个 `PaymentStrategy` 接口，无论用户选哪个，业务层都是统一调用 `Pay()` 方法。其实你的 `AIModel` 接口（可以随时替换底层模型）也蕴含了强烈的策略模式思想。
- **观察者模式 (Observer / Pub-Sub)**：
	- **核心思想**：B 站 up 主发视频，所有关注他的粉丝都会自动收到推送通知。一处数据改变，多处自动响应。
	- **实战**：你在架构设计中可能会用到的消息队列（如 RabbitMQ 或 Kafka）。订单服务创建了订单（发布者），库存服务和短信服务（订阅者）监听到事件后各自执行扣库存和发短信的操作，实现微服务间的异步解耦。

#### 设计一个懒汉的单例模式

手撕一个 `AIModelFactory` 的懒汉式单例

```go
package main

import(
	"sync"
)
type AIModelFactory struct{
    creators map[string]ModelCreator
}
var(
	instance  *AIModelFactory
    once	   sync.Once
)
func GetGlobalFactory() *AIModelFactory{
    once.Do(func()){
        instance = &AIModelFactory{
             creators: make(map[string]ModelCreator),
        }
    }
    return instance
}
```

面试官您好，在 Go 语言中实现懒汉式单例，我首选标准库提供的 `sync.Once`。

首先，我声明了私有的 `instance` 指针和 `once` 变量，防止外部直接修改。

在 `GetGlobalFactory()` 方法中，调用了 `once.Do()`。
