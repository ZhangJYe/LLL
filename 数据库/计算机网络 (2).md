#### [⭐️TCP/IP 四层模型是什么？每一层的作用是什么？](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️tcp-ip-四层模型是什么-每一层的作用是什么)
面试官您好。TCP/IP 四层模型是现代互联网通信的基础架构，它将网络通信从上到下分为了四层，分别是：

**应用层、传输层、网络层和网络接口层**。

1. **应用层：** 离我们开发者最近，负责处理特定的应用程序细节，为用户提供服务。我们后端常打交道的 HTTP、HTTPS、RPC、DNS 等协议都在这一层。
2. **传输层：** 负责为两台主机上的应用程序提供端到端的通信。它利用**端口号**来区分不同的应用。最核心的协议是面向连接、可靠的 TCP，以及无连接、高效的 UDP。
3. **网络层：** 负责数据包在复杂网络中的路由和转发，决定数据从源到目的地的路径。它使用 **IP 地址** 来定位网络中的不同主机。核心协议是 IP 协议。
4. **网络接口层：** 这是最底层，负责与底层的物理硬件交互，将数据封装成帧并通过物理网络（如以太网、Wi-Fi）发送出去，它依赖的是硬件的 **MAC 地址**。

总的来说，数据发送时是自上而下逐层封装，接收时是自下而上逐层解包。

当 Go 服务发送一个 HTTP 响应时，数据会从**应用层**出发，逐层向下打包（加上 TCP 头、IP 头、以太网帧头），变成电信号发送出去；接收端收到后，再自底向上逐层解包，最终交还给应用程序。

#### 应用层有哪些常见的协议

+ **HTTP（Hypertext Transfer Protocol，超文本传输协议）**：  基于 TCP 协议，是一种用于**传输超文本和多媒体内容的协议**，主要是为 Web 浏览器与 Web 服务器之间的通信而设计的。当我们使用浏览器浏览网页的时候，我们网页就是通过 HTTP 请求进行加载的。
+ **SMTP（Simple Mail Transfer Protocol，简单邮件发送协议）**：  基于 TCP 协议，是一种用于发送电子邮件的协议。注意 ⚠️：SMTP 协议只负责邮件的发送，而不是接收。要从邮件服务器接收邮件，需要使用 POP3 或 IMAP 协议。
+ **POP3/IMAP（邮件接收协议）**：  基于 TCP 协议，两者都是负责邮件接收的协议。IMAP 协议是比 POP3 更新的协议，它在功能和性能上都更加强大。IMAP 支持邮件搜索、标记、分类、归档等高级功能，而且可以在多个设备之间同步邮件状态。几乎所有现代电子邮件客户端和服务器都支持 IMAP。
+ **FTP（File Transfer Protocol，文件传输协议）** :   基于 TCP 协议，是一种用于在计算机之间传输文件的协议，可以屏蔽操作系统和文件存储方式。注意 ⚠️：FTP 是一种不安全的协议，因为它在传输过程中不会对数据进行加密。建议在传输敏感数据时使用更安全的协议，如 SFTP。
+ **Telnet（远程登陆协议）**：  基于 TCP 协议，用于通过一个终端登陆到其他服务器。Telnet 协议的最大缺点之一是所有数据（包括用户名和密码）均以明文形式发送，这有潜在的安全风险。这就是为什么如今很少使用 Telnet，而是使用一种称为 SSH 的非常安全的网络传输协议的主要原因。
+ **SSH（Secure Shell Protocol，安全的网络传输协议）**：  基于 TCP 协议，通过加密和认证机制实现安全的访问和文件传输等业务
+ **RTP（Real-time Transport Protocol，实时传输协议）**：  通常基于 UDP 协议，但也支持 TCP 协议。它提供了端到端的实时传输数据的功能，但不包含资源预留存、不保证实时传输质量，这些功能由 WebRTC 实现。
+ **DNS（Domain Name System，域名管理系统）**:   基于 UDP 协议，用于解决域名和 IP 地址的映射问题。

#### 传输层常见的协议
+ **TCP（Transmission Control Protocol，传输控制协议 ）**：提供 **面向连接** 的，**可靠** 的数据传输服务。
+ **UDP（User Datagram Protocol，用户数据协议）**：提供 **无连接** 的，**尽最大努力** 的数据传输服务（不保证数据传输的可靠性），简单高效。



**QUIC（Quick UDP Internet Connections）  ： 最初是由 Google 开发的，现在已经成为了 HTTP/3 的底层传输协议。虽然它的名字里有 UDP，但它其实是一个伪装成 UDP 的 TCP+TLS。  **

它之所以被创造出来，主要是为了解决 TCP 的历史包袱。

因为 TCP 是实现在操作系统内核里的，升级极其缓慢。

所以 QUIC 另辟蹊径，**直接在用户态基于 UDP 构建。**

它既保留了 UDP 快速、无连接的特性，又在应用层实现了类似 TCP 的丢包重传、拥塞控制，并且**内置了 TLS 1.3 的加密。**

**它最核心的优势有两个：**

**一是解决了 TCP 的队头阻塞（Head-of-Line Blocking）问题，即使丢包也只会影响当前的数据流，不会阻塞整个连接；**

**二是极大地降低了建连延迟，甚至可以做到 0-RTT（零往返时间）的数据发送，这对于弱网环境和移动端来说是革命性的提升。”**

#### 网络层常见协议

![](https://javaguide.cn/assets/nerwork-layer-protocol-VpGZIByy.png)

+ **IP（Internet Protocol，网际协议）**：TCP/IP 协议中最重要的协议之一，属于网络层的协议，主要作用是定义数据包的格式、对数据包进行路由和寻址，以便它们可以跨网络传播并到达正确的目的地。目前 IP 协议主要分为两种，一种是过去的 IPv4，另一种是较新的 IPv6，目前这两种协议都在使用，但后者已经被提议来取代前者。
+ **ARP（Address Resolution Protocol，地址解析协议）**：ARP 协议解决的是**网络层地址和链路层地址**之间的转换问题。因为一个 IP 数据报在物理上传输的过程中，总是需要知道下一跳（物理上的下一个目的地）该去往何处，但 IP 地址属于逻辑地址，而 MAC 地址才是物理地址，ARP 协议解决了 IP 地址转 MAC 地址的一些问题。
+ **ICMP（Internet Control Message Protocol，互联网控制报文协议）**：一种用于传输网络状态和错误消息的协议，常用于网络诊断和故障排除。例如，Ping 工具就使用了 ICMP 协议来测试网络连通性。
+ **NAT（Network Address Translation，网络地址转换协议）**：NAT 协议的应用场景如同它的名称——网络地址转换，应用于内部网到外部网的地址转换过程中。具体地说，在一个小的子网（局域网，LAN）内，各主机使用的是同一个 LAN 下的 IP 地址，但在该 LAN 以外，在广域网（WAN）中，需要一个统一的 IP 地址来标识该 LAN 在整个 Internet 上的位置。
+ **OSPF（Open Shortest Path First，开放式最短路径优先）**：一种内部网关协议（Interior Gateway Protocol，IGP），也是广泛使用的一种动态路由协议，基于链路状态算法，考虑了链路的带宽、延迟等因素来选择最佳路径。
+ **RIP(Routing Information Protocol，路由信息协议）**：一种内部网关协议（Interior Gateway Protocol，IGP），也是一种动态路由协议，基于距离向量算法，使用固定的跳数作为度量标准，选择跳数最少的路径作为最佳路径。
+ **BGP（Border Gateway Protocol，边界网关协议）**：一种用来在路由选择域之间交换网络层可达性信息（Network Layer Reachability Information，NLRI）的路由选择协议，具有高度的灵活性和可扩展性。

### 三.HTTP
#### [<font style="background-color:#FBDE28;">⭐</font><font style="background-color:#FBDE28;">️从输入 URL 到页面展示到底发生了什么？（非常重要）</font>](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️从输入-url-到页面展示到底发生了什么-非常重要)
面试官您好，从输入 URL 到页面展示，整个过程大致可以分为以下几个核心阶段：

1.**URL 解析与缓存查找：** 

首先浏览器会解析输入的 URL，判断是合法的 URL 还是搜索关键词。

接着，浏览器会依次检查**浏览器缓存、操作系统缓存、路由器缓存，看是否有该域名的 IP 记录。**

2.**DNS 解析：** 

	如果缓存中没有，就会向本地 DNS 服务器发起请求，通过递归和迭代查询，最终获取到域名对应的目标 IP 地址。

3.**建立 TCP 连接（三次握手）：** 

	拿到 IP 后，浏览器会通过操作系统的网络协议栈，向服务器的 80（HTTP）或 443（HTTPS）端口发起 TCP 连接。这中间会经历经典的‘三次握手’来确保连接的可靠性。如果是 HTTPS，还会在 TCP 连接之后进行 TLS/SSL 握手，协商加密密钥。

4.**发送 HTTP/HTTPS 请求：** 

	连接建立后，浏览器会构建 HTTP 请求报文（包含请求行、请求头、请求体），并发送给服务端。

5.**服务器处理请求并返回响应：** 

	服务端接收到请求后（通常会经过负载均衡、网关），会将**请求路由给具体的后端服务**。后端服务**执行业务逻辑**（比如查库、计算），**生成 HTTP 响应报文**（包含状态码、响应头、响应体，如 HTML 页面或 JSON 数据），然后返回给浏览器。

6.**浏览器渲染页面：**

	 浏览器拿到 HTML 内容后，会解析 DOM 树和 CSSOM 树，合并成渲染树，然后计算布局并绘制到屏幕上。遇到 JS、图片等静态资源会继续发起网络请求去获取。

7.**断开 TCP 连接（四次挥手）：** 

	数据传输完毕后，通常会根据 `Connection: keep-alive` 的策略，在合适的时机通过‘四次挥手’断开连接。



在实际的微服务架构中，比如构建一个带有弹幕功能的视频网站后台，第 5 步实际上是非常复杂的。

流量打到服务器后，首先会经过一层**反向代理**（如 Nginx），然后到达 API 网关。

网关可能会根据协议**将 HTTP 请求转换为 RPC 调用**。接着，像使用 Kratos 这样的微服务框架编写的后端服务，会进行鉴权、限流，然后去调用具体的下游服务（比如用户服务、视频元数据服务、弹幕服务）。

这些服务可能还会涉及 Redis 缓存命中、MySQL 数据库查询，甚至涉及到分布式系统中的一致性处理（比如**底层数据复制用到 Raft 算法**）。

最终将各个服务的数据聚合后，序列化成 JSON 返回给客户端。

#### 你提到了微服务中的 RPC 调用，你们内部服务通信为什么选 RPC 而不是直接用 HTTP？
面试官您好。在微服务内部通信中，我们通常优先选择 RPC（特别是像 gRPC 这样的方案）而不是直接使用传统的 HTTP/RESTful API，主要出于以下三个维度的考量：

1. **传输效率与性能更高：**  传统的 HTTP 接口通常基于 HTTP/1.1 协议，**数据载体多为 JSON**，这是一种文本格式，体积大且序列化/反序列化消耗 CPU。 而现代 RPC（如 gRPC）通常基于 HTTP/2 协议，支持多路复用，并且使用 Protobuf（Protocol Buffers）这种**二进制序列化协议**。二进制数据体积更小，网络传输更快，解析效率也远高于 JSON。
2. **契约优先与代码自动生成（IDL）：** 使用 RPC 时，我们通常会通过 IDL（接口定义语言，如 `.proto` 文件）来严格定义服务间交互的输入输出结构。 依靠工具可以直接生成强类型的客户端和服务端代码。这不仅规范了接口，还避免了手写 HTTP 请求时容易出现的字段拼写错误、类型不匹配等问题，降低了微服务间的沟通成本。
3. **调用方式更自然（语义化）：**  RPC（远程过程调用）的核心思想是屏蔽网络通信细节。 在代码层面，调用远程服务就像调用本地函数一样自然（例如 `userService.GetUser(ctx, req)`），开发者不需要去关心底层的 URL 拼接、HTTP 方法（GET/POST）以及各种 HTTP 状态码的管理，能更专注在业务逻辑上。

当然，对外的网关接口或者与前端/客户端交互时，我们依然会提供 HTTP/RESTful API，因为它的通用性和跨平台性更好。但在内部高频、低延迟的服务间调用场景，RPC 是更优的解。

##### 你刚才提到 gRPC 用了 Protocol Buffers，它为什么比 JSON 快那么多？底层是怎么压缩数据的？
面试官您好。gRPC 默认使用的 Protobuf 之所以比 JSON 快得多，核心在于它在**序列化和反序列化**时，做到了**体积更小**和**解析更快**。

具体体现在以下三个底层机制：

1. **省略字段名（Key），依靠 Schema 映射：**  JSON 是自描述的文本，每次传输都会带上冗长的字段名（比如 `"username": "张三"`）。 而 Protobuf 在传输时不传字段名，只传一个**字段编号（Tag）**和对应的值。双方通过预先定义好的 `.proto` 文件作为‘字典’来对应数据，这极大地减少了网络传输的冗余字节。
2. **二进制格式，无需文本解析：**  JSON 是纯文本（字符串），程序在解析时需要逐个字符扫描（比如寻找大括号、引号、冒号），处理字符集编码，并将字符串转换成内存中的整型或浮点型，极其消耗 CPU。 而 Protobuf 是紧凑的二进制字节流，机器可以直接按位读取，跳过了复杂的文本词法分析。
3. **底层的核心压缩算法（Varint 与 ZigZag）：**  对于数字的存储，Protobuf 并没有死板地使用定长的 4 字节（int32）或 8 字节（int64）。 它使用了 **Varint（变长整数编码）**，数值越小，占用的字节数越少，比如数字 `1` 只需要 1 个字节就能存下。对于负数，它还配合使用了 **ZigZag 编码**，将有符号数映射为无符号数，再用 Varint 压缩，避免了负数补码导致的字节浪费。

综上所述，传输数据量小加上不需要文本级的词法分析，造就了 Protobuf 极高的性能表现。

另外，在 Go 语言的实现层面也有巨大差异。

Go 标准库的 `encoding/json` 在反序列化时严重依赖**反射（Reflection）**，这需要**在运行时动态获取结构体的类型信息**，不仅慢，还会产生大量的堆内存分配，给 GC（垃圾回收）带来压力。

而 Protobuf 通过 `protoc-gen-go` 插件，在编译期就已经生成了具体的强类型序列化/反序列化代码，运行时直接按位操作，几乎避开了反射带来的开销。

#### grpc的序列化和反序列化是什么？
简单来说，**序列化（Serialization）****就是把我们在程序里运行的复杂数据结构（比如 Golang 里的一个包含很多字段的 Struct），转换成一段可以放在网络上传输的‘二进制字节流’；**

**而反序列化（Deserialization）**，就是接收端拿到这段字节流后，再把它重新还原成程序能看懂的、方便操作的数据结构。



你可以把它想象成**搬家时拆装家具**：内存里的对象就是一个拼装好的大衣柜，你没法直接把它塞进网络这根‘细水管’里。所以你需要先把它拆成一块块的木板打包（序列化为二进制流），运到新机器后，再按照图纸重新组装成衣柜（反序列化为对象）。

在 gRPC 的世界里，承担这个‘拆装’工作的核心主角是 **Protocol Buffers（简称 Protobuf）**。

它抛弃了传统的 JSON 那种人类可读的文本格式，直接采用极其紧凑的二进制编码，这也正是 gRPC 速度极快的原因所在。

#### ✨微服务多了之后，服务之间是怎么找到对方的？也就是服务注册与发现（比如用 Consul、Etcd），或者底层的一致性算法（比如 Raft）是怎么工作的？
面试官您好。

在微服务架构中，服务之间的寻址主要依靠**服务注册与发现**机制，而像 Etcd 这样的注册中心底层则是通过 **Raft 一致性算法**来保证高可用的。

**第一部分：服务注册与发现是怎么工作的？**

1. **服务注册 (Register)：**  当一个服务提供者（Provider）启动时，它会将自己的**网络地址（IP 和端口）**、**服务名等元数据**发送给注册中心（如 Etcd 或 Consul），注册中心会把这些信息存下来。
2. **健康检查 (Heartbeat)：**  服务提供者会**定期向注册中心发送心跳**。如果注册中心在规定时间内没收到心跳，就会将该实例从可用列表中剔除。
3. **服务发现 (Discover/Watch)：**  当服务消费者（Consumer）需要调用其他服务时，它会向注册中心**查询目标服务的实例列表**。 为了提高效率，消费者通常会将列表缓存在本地，并通过长连接监听（Watch机制）注册中心的变更事件。 一旦有实例上下线，消费者能立刻更新本地缓存，然后结合负载均衡策略（如轮询、随机）发起内部 RPC 调用。

**第二部分：底层的 Raft 算法是怎么工作的？** 

Etcd 等组件能可靠地存储服务信息，依赖于底层的 Raft 算法。

Raft 主要通过两个核心机制来保证分布式集群的数据一致性：

1. **领导者选举 (Leader Election)：**  Raft 集群中的节点有三种状态：**Leader（领导者）、Follower（跟随者）和 Candidate（候选者）**。 正常情况下只有 Leader 处理写请求并向 Follower 发送心跳。 如果 Follower 超时未收到心跳，就会转为 Candidate 发起选举，获得多数派选票的节点成为新 Leader。
2. **日志复制 (Log Replication)：**  当客户端发起写请求时，Leader 首先将请求作为日志条目写入本地，然后并行发送给所有 Follower。 当半数以上的 Follower 回复已接收后，Leader 就会将这条日志‘提交’（Commit）并应用到状态机，最后向客户端返回成功，并通知 Follower 也提交该日志。

简单来说，服务发现解决了微服务‘你在哪’的问题，而 Raft 算法解决了注册中心集群自身‘数据一致不丢失’的问题。”

面试官您好。

HTTP 状态码由三个十进制数字组成，主要**用来表示客户端 HTTP 请求的返回结果**。

它按照首位数字分为了五大类：

1. **1xx（信息性状态码）：** **表示接收的请求正在处理**。开发中比较少见，常见的如 **101 Switching Protocols**，在升级为 WebSocket 协议时会用到。
2. **2xx（成功状态码）：** **表示请求正常处理完毕**。
    - **200 OK：** 最常见，请求成功且有数据返回。
    - **201 Created：** 通常在 RESTful API 中表示 POST 请求成功创建了新资源。
    - **204 No Content：** 请求成功，但没有实体的主体返回（比如成功的 DELETE 请求）。
3. **3xx（重定向状态码）：** 表示**需要客户端采取进一步操作才能完成请求**。
    - **301 Moved Permanently：** 永久重定向，搜索引擎会更新索引。
    - **302 Found：** 临时重定向。
    - **304 Not Modified：** 资源未修改，告诉浏览器直接使用本地缓存（这在静态资源优化中非常重要）。
4. **4xx（客户端错误状态码）：** 表示**客户端发出的请求有误**。
    - **400 Bad Request：** 请求报文存在语法错误或参数缺失。
    - **401 Unauthorized：** 缺少认证信息（如未登录或 Token 失效）。
    - **403 Forbidden：** 认证通过了，但没有权限访问该资源。
    - **404 Not Found：** 请求的资源不存在。
    - **429 Too Many Requests：** 客户端请求过于频繁，通常触发了后端的限流策略。
5. **5xx（服务器错误状态码）：** 表示**服务器处理请求时发生了内部错误**。
    - **500 Internal Server Error：** 服务端代码报错（比如代码里 panic 且没被 recover，或者出现了预期外的异常）。
    - **502 Bad Gateway：** 作为网关或代理（如 Nginx）工作时，从上游的后端服务拿到了无效响应（通常是后端服务挂了）。
    - **504 Gateway Timeout：** 网关超时，说明上游的后端服务还在运行，但处理太慢，导致 Nginx 等不及了。

总结来说，1、2、3 开头通常代表**请求顺利或需要端侧配合**，4 开头需要**检查前端代码和传参**，5 开头则是我们**后端程序员需要马上排查报警的故障。**

#### HTTP Header 中常见的字段有哪些？
面试官您好，HTTP Header 包含了大量的控制信息，为了便于理解，我通常将它们分为以下几个核心类别：

1. **内容描述类 (Content / Entity)：** 告诉接收方发来的是什么数据。
    - `Content-Type`：极为重要，比如我们做后端 API 最常用的 `application/json`，或者上传文件用的 `multipart/form-data`。
    - `Content-Length`：表示 Body 的字节长度，用来防止粘包或者截断。
2. **身份认证与状态类 (Auth & Session)：**
    - `Authorization`：常用来携带 Token（比如 JWT），格式通常是 `Bearer <token>`，这是微服务鉴权的核心。
    - `Cookie` / `Set-Cookie`：早期的 Web 状态管理，前端带着 Cookie 来，后端用 Set-Cookie 种下去。
3. **客户端/服务端信息类 (Context)：**
    - `Host`：指定请求的服务器域名，这是 HTTP/1.1 必须带的字段。
    - `User-Agent`：包含了发起请求的浏览器、操作系统版本等信息，后端常用来做简单的反爬虫或埋点统计。
    - `Server`：响应头中说明服务端的软件信息（如 `nginx/1.18.0`，出于安全考虑通常会隐藏版本号）。
4. **连接与缓存控制类 (Connection & Cache)：**
    - `Connection`：常用 `keep-alive`，告诉对方保持 TCP 连接，不要马上断开，以复用连接。
    - `Cache-Control`：非常复杂且重要的缓存策略，比如 `no-cache`、`max-age`，配合 `ETag` 或 `Last-Modified` 控制浏览器缓存。
5. **跨域控制类 (CORS)：**
    - `Access-Control-Allow-Origin`：响应头，用来告诉浏览器允许哪些域名前来跨域请求，这是前后端分离架构中最常打交道的字段。

总的来说，熟练掌握这些 Header 字段，是后端开发排查网络问题、设计健壮 API 的基本功。

#### [⭐️HTTP 和 HTTPS 有什么区别？（重要）](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️http-和-https-有什么区别-重要)
<!-- 这是一张图片，ocr 内容为： -->
![](https://oss.javaguide.cn/github/javaguide/cs-basics/network/http-vs-https.png)

面试官您好，HTTP 和 HTTPS 的核心区别主要体现在**安全性、默认端口、证书需求以及连接建立的开销**这四个方面：

1. **安全性（核心区别）：**  HTTP 是超文本传输协议，数据在网络中是**明文传输**的，极易被中间人窃听、篡改或劫持。 而 HTTPS（HyperText Transfer Protocol Secure）是在 HTTP 的下层加入了 **SSL/TLS 协议**。 它通过**加密传输和身份认证**，保证了数据的机密性和完整性。
2. **默认端口不同：**  HTTP 默认使用的是 **80** 端口，而 HTTPS 默认使用的是 **443** 端口。
3. **证书需求：**  HTTPS 协议需要向 CA（证书颁发机构）申请并配置数字证书，用来证明服务器的身份，防止客户端访问到伪造的钓鱼网站。 而 HTTP 不需要。
4. **连接建立的过程与开销：**  HTTP 只需要进行 TCP 的三次握手就可以开始传输数据。 而 HTTPS 在 TCP 三次握手之后，还需要进行复杂的 **TLS 握手**（包括协商加密算法、交换密钥、验证证书等），建立起安全的加密信道后，才能开始传输原本的 HTTP 数据。 因此，HTTPS 的握手延迟和 CPU 计算开销会比 HTTP 略大。

总结来说，HTTPS 就是‘披着 SSL/TLS 外壳的 HTTP’，牺牲了一点点连接建立的性能，换来了极大的安全性。



在实际的 Go 语言开发中，启动一个 HTTP 服务我们通常用 http.ListenAndServe(":80", nil)，而**启动 HTTPS 服务**则需要用到 http.ListenAndServeTLS(":443", "cert.pem", "key.pem", nil)，必须**传入证书和私钥文件**。

不过在现代的微服务架构中（比如使用 Go-Zero 或 Kratos 框架构建的系统），我们很少在具体的业务微服务节点上直接配置 HTTPS。

通常的做法是：在最外层的 API 网关（如 Nginx、Envoy）统一做 TLS 卸载（TLS Termination）。

也就是说，客户端到网关这一段走 HTTPS 保证公网安全；

网关解密后，在公司内网（VPC）中，网关到各个 Go 微服务之间，或者微服务互相进行 RPC 调用时，为了追求极致的性能，往往会降级走纯内网的 HTTP 或明文 gRPC（除非金融级系统要求全链路加密）。

#### [HTTP/1.0 和 HTTP/1.1 有什么区别？](https://javaguide.cn/cs-basics/network/other-network-questions.html#http-1-0-和-http-1-1-有什么区别)
面试官您好，HTTP/1.0 和 HTTP/1.1 的区别主要体现在**连接管理、Host 头处理、带宽优化以及缓存控制**这四个核心方面：

1. **长连接 (Keep-Alive) 默认开启（最核心区别）：**
    1.  **HTTP/1.0 默认使用短连接**，每次发起 HTTP 请求都要**重新经历 TCP 的三次握手和四次挥手**，开销极大。
    2.  而 **HTTP/1.1 默认开启了长连接**（`Connection: keep-alive`），允许在同一个 TCP 连接上复用、发送多个 HTTP 请求和响应，大幅降低了网络延迟。
2. **强制要求 Host 头（支持虚拟主机）：**  
    1. 在 HTTP/1.0 时代，认为一个服务器只有一个 IP，绑定一个站点。但随着互联网发展，一台物理服务器通常会托管多个域名的网站。
    2. HTTP/1.1 强制请求消息中必须包含 `Host` 请求头，否则会返回 400 状态码，这使得基于域名的虚拟主机成为了可能。
3. **带宽优化与断点续传（Range 请求）：**  
    1. HTTP/1.0 只能一次性请求完整的对象。
    2. HTTP/1.1 引入了 `Range` 请求头，允许客户端只请求资源的一部分（比如返回 `206 Partial Content` 状态码）。
    3. 这为我们实现文件断点续传、分块传输（Chunked Transfer Encoding）提供了底层支持。
4. **更强大的缓存控制：**  
    1. HTTP/1.0 主要是靠 `Expires`（绝对时间）和 `If-Modified-Since` 来控制缓存。
    2. HTTP/1.1 引入了更加灵活和细粒度的 `Cache-Control`（如 `max-age` 相对时间），以及基于内容哈希的 `ETag` 机制，让缓存策略更加可靠。”



在 Go 语言的 net/http 标准库中，客户端（Client）默认就是基于 HTTP/1.1 并且开启了 Keep-Alive 的。

底层通过 http.Transport 维护了一个 TCP 连接池（可以通过 MaxIdleConns 和 MaxIdleConnsPerHost 来调优）。

这里有一个非常经典的 Go 语言新手坑：如果我们在发起 HTTP 请求后，没有正确读取并关闭响应体 (**defer resp.Body.Close()**)，Go 的底层 transport 就无法将这个 TCP 连接放回连接池复用。

这不仅会导致 Keep-Alive 失效退化成短连接，高并发下还会直接导致文件句柄（Socket）泄露，最终把服务器打挂。

#### [⭐️HTTP/1.1 和 HTTP/2.0 有什么区别？](https://javaguide.cn/cs-basics/network/other-network-questions.html#%E2%AD%90%EF%B8%8Fhttp-1-1-%E5%92%8C-http-2-0-%E6%9C%89%E4%BB%80%E4%B9%88%E5%8C%BA%E5%88%AB)
+ **多路复用（Multiplexing）**：
    - HTTP/2.0 在**同一连接上可以同时传输多个请求和响应**（可以看作是 HTTP/1.1 中长链接的升级版本），互不干扰。
    - HTTP/1.1 则使用**串行方式**，每个请求和响应都需要独立的连接，而浏览器为了控制资源会有 6-8 个 TCP 连接的限制。
    - 这使得 HTTP/2.0 在处理多个请求时更加高效，减少了网络延迟和提高了性能。
+ **二进制帧（Binary Frames）**：
    - HTTP/2.0 使用**二进制帧**进行数据传输，而 HTTP/1.1 则使用**文本格式**的报文。
    - 二进制帧更加紧凑和高效，减少了传输的数据量和带宽消耗。
+ **队头阻塞**：
    - HTTP/2 引入了**多路复用技术**，允许多个请求和响应在单个 TCP 连接上并行交错传输，解决了 HTTP/1.1 应用层的队头阻塞问题，但 HTTP/2 依然受到 TCP 层队头阻塞 的影响。
+ **头部压缩（Header Compression）**：
    - HTTP/1.1 支持`Body`压缩，`Header`不支持压缩。
    - HTTP/2.0 支持对`Header`压缩，使用了专门为`Header`压缩而设计的 HPACK 算法，减少了网络开销。
+ **服务器推送（Server Push）**：HTTP/2.0 支持服务器推送，可以在客户端请求一个资源时，将其他相关资源一并推送给客户端，从而减少了客户端的请求次数和延迟。而 HTTP/1.1 需要客户端自己发送请求来获取相关资源。

#### [HTTP/2.0 和 HTTP/3.0 有什么区别？](https://javaguide.cn/cs-basics/network/other-network-questions.html#http-2-0-%E5%92%8C-http-3-0-%E6%9C%89%E4%BB%80%E4%B9%88%E5%8C%BA%E5%88%AB)
+ **<font style="color:rgb(60, 60, 67);">传输协议</font>**<font style="color:rgb(60, 60, 67);">：</font>
    - <font style="color:rgb(60, 60, 67);">HTTP/2.0 是基于 TCP 协议实现的，HTTP/3.0 新增了 QUIC（Quick UDP Internet Connections） 协议来实现可靠的传输，提供与 TLS/SSL 相当的安全性，具有较低的连接和传输延迟。</font>
    - <font style="color:rgb(60, 60, 67);">你可以将 QUIC 看作是 UDP 的升级版本，在其基础上新增了很多功能比如加密、重传等等。HTTP/3.0 之前名为 HTTP-over-QUIC，从这个名字中我们也可以发现，HTTP/3 最大的改造就是使用了 QUIC。</font>
+ **<font style="color:rgb(60, 60, 67);">连接建立</font>**<font style="color:rgb(60, 60, 67);">：</font>
    - <font style="color:rgb(60, 60, 67);">HTTP/2.0 需要经过经典的 TCP 三次握手过程（由于安全的 HTTPS 连接建立还需要 TLS 握手，共需要大约 3 个 RTT）。</font>
    - <font style="color:rgb(60, 60, 67);">由于 QUIC 协议的特性（TLS 1.3，TLS 1.3 除了支持 1 个 RTT 的握手，还支持 0 个 RTT 的握手）连接建立仅需 0-RTT 或者 1-RTT。这意味着 QUIC 在最佳情况下不需要任何的额外往返时间就可以建立新连接。</font>
+ **<font style="color:rgb(60, 60, 67);">头部压缩</font>**<font style="color:rgb(60, 60, 67);">：HTTP/2.0 使用 HPACK 算法进行头部压缩，而 HTTP/3.0 使用更高效的 QPACK 头压缩算法。</font>
+ **<font style="color:rgb(60, 60, 67);">队头阻塞</font>**<font style="color:rgb(60, 60, 67);">：HTTP/2.0 多请求复用一个 TCP 连接，一旦发生丢包，就会阻塞住所有的 HTTP 请求。由于 QUIC 协议的特性，HTTP/3.0 在一定程度上解决了队头阻塞（Head-of-Line blocking, 简写：HOL blocking）问题，一个连接建立多个不同的数据流，这些数据流之间独立互不影响，某个数据流发生丢包了，其数据流不受影响（本质上是多路复用+轮询）。</font>
+ **<font style="color:rgb(60, 60, 67);">连接迁移</font>**<font style="color:rgb(60, 60, 67);">：HTTP/3.0 支持连接迁移，因为 QUIC 使用 64 位 ID 标识连接，只要 ID 不变就不会中断，网络环境改变时（如从 Wi-Fi 切换到移动数据）也能保持连接。而 TCP 连接是由（源 IP，源端口，目的 IP，目的端口）组成，这个四元组中一旦有一项值发生改变，这个连接也就不能用了。</font>
+ **<font style="color:rgb(60, 60, 67);">错误恢复</font>**<font style="color:rgb(60, 60, 67);">：HTTP/3.0 具有更好的错误恢复机制，当出现丢包、延迟等网络问题时，可以更快地进行恢复和重传。而 HTTP/2.0 则需要依赖于 TCP 的错误恢复机制。</font>
+ **<font style="color:rgb(60, 60, 67);">安全性</font>**<font style="color:rgb(60, 60, 67);">：在 HTTP/2.0 中，TLS 用于加密和认证整个 HTTP 会话，包括所有的 HTTP 头部和数据负载。TLS 的工作是在 TCP 层之上，它加密的是在 TCP 连接中传输的应用层的数据，并不会对 TCP 头部以及 TLS 记录层头部进行加密，所以在传输的过程中 TCP 头部可能会被攻击者篡改来干扰通信。而 HTTP/3.0 的 QUIC 对整个数据包（包括报文头和报文体）进行了加密与认证处理，保障安全性。</font>

#### [⭐️HTTP 是不保存状态的协议, 如何保存用户状态?](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️http-是不保存状态的协议-如何保存用户状态)
面试官您好。

由于 HTTP 是无状态的，为了在多次请求中识别用户身份、保存用户状态，业界目前主要有两种主流方案：

**基于 Cookie-Session 的服务端状态方案** 和 **基于 Token（如 JWT）的客户端状态方案**。

1. **方案一：Cookie + Session（传统 Web 开发常用）**
    - **原理：** 用户登录成功后，服务端会在内存或数据库（如 Redis）中**创建一个 Session 对象来保存用户信息**，并生成一个唯一的 `SessionID`。然后，服务端通过 HTTP 响应头（`Set-Cookie`）将这个 `SessionID` 种到客户端的浏览器中。
    - **后续请求：** 客户端每次发请求时，浏览器会自动在 HTTP 请求头（`Cookie`）中带上这个 `SessionID`。服务端拿到 ID 后，去 Redis 里查出对应的 Session，从而“认出”这个用户。
2. **方案二：Token / JWT (JSON Web Token)（前后端分离与微服务首选）**
    - **原理：** 用户登录成功后，**服务端不保存任何 Session 状态**，而是**将用户信息（如 UserID、角色）和一个数字签名打包成一个加密的 Token 字符串**，直接下发给客户端。
    - **后续请求：** **客户端（如前端应用、App）把 Token 存起来**，并在每次请求的 HTTP Header（通常是 `Authorization: Bearer <token>`）中主动携带它。服务端收到 Token 后，只需要通过预先配置的**密钥**去校验签名的合法性，就能解析出用户信息。

**总结比较：**

Session 方案状态存在服务端，安全性高且容易强制用户下线，但不利于服务器水平扩展；

JWT 方案状态存在客户端，服务端无状态（Stateless），非常适合分布式系统和微服务架构，但一旦签发在过期前较难主动撤销。



在实际的开发中，绝大多数情况会选择 JWT。

因为微服务往往是多节点部署的。

如果用 Session，请求打到 A 节点生成了 Session，下一次请求如果被负载均衡路由到了 B 节点，B 节点内存里没有这个 Session，用户就会被迫重新登录（这叫 Session 共享问题，虽然可以用 Redis 集中存储解决，但也增加了系统的耦合）。

而 JWT 的优势在于去中心化。

用户服务签发 JWT 后，弹幕服务、视频元数据服务在收到 gRPC 调用时，只需要用公钥或共享密钥验证一下 JWT 的签名，就知道这是哪个用户了，完全不需要去查数据库或 Redis，极大地降低了内部网络开销。

#### [URI 和 URL 的区别是什么?](https://javaguide.cn/cs-basics/network/other-network-questions.html#uri-和-url-的区别是什么)
+ URI(Uniform Resource Identifier) 是**统一资源标志符**，可以唯一标识一个资源。
+ URL(Uniform Resource Locator) 是**统一资源定位符**，可以**提供该资源的路径**。它是一种具体的 URI，即 URL 可以用来标识一个资源，而且还指明了如何 locate 这个资源。

URI 的作用像身份证号一样，URL 的作用更像家庭住址一样。

#### [⭐️GET 和 POST 的区别](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️get-和-post-的区别)
面试官您好。

GET 和 POST 是 HTTP 协议中最常用的两种请求方法，它们的核心区别主要体现在**语义规范、参数传递、缓存机制以及幂等性**这四个维度：

1. **语义与用途（最核心的区别）：**
    - **GET** 的语义是**获取资源**。它是“安全”的（不修改服务器状态），主要用于查询数据。
    - **POST** 的语义是**提交数据**。通常用于创建新资源或触发改变服务器状态的操作。
2. **参数传递位置与限制：**
    - GET 请求的参数通常放在 URL 中，形成查询字符串（querystring）
    - 而 POST 请求的参数通常放在请求体（body）中，可以有多种编码格式，如 application/x-www-form-urlencoded、multipart/form-data、application/json 等。
    - GET 请求的 URL 长度受到浏览器和服务器的限制，而 POST 请求的 body 大小则没有明确的限制。
    - 不过，实际上 GET 请求也可以用 body 传输数据，只是并不推荐这样做，因为这样可能会导致一些兼容性或者语义上的问题。
3. **缓存机制：**
    - **GET** 请求默认是可以被浏览器、CDN、甚至网关缓存的。
    - **POST** 请求默认不会被缓存，每次都会打到后端的业务服务上。
4. **幂等性（Idempotency）：**
    - **GET 是幂等的**。无论调用多少次，对系统资源产生的影响都是一样的。
    - **POST 不是幂等的**。比如一个非幂等的创建订单接口，如果因为网络卡顿导致客户端重试发送了两次 POST 请求，服务器可能就会创建两个完全一样的订单。

总的来说，GET 用来读，POST 用来写，这是遵循 REST API 设计规范的基础。

### 四.Websocket
#### 什么是websocket
WebSocket 是一种基于 TCP 连接的全双工通信协议，即**客户端和服务器可以同时发送和接收数据。**

WebSocket 协议本质上是应用层的协议，用于弥补 HTTP 协议在持久通信能力上的不足。客户端和服务器仅需一次握手，两者之间就直接可以创建持久性的连接，并进行双向数据传输。

它的工作流程通常是这样的：

1. **握手（Handshake）：** 客户端首先发起一个普通的 HTTP 请求，但在请求头中带上 `Upgrade: websocket` 和 `Connection: Upgrade` 字段，告诉服务器‘我想把协议升级成 WebSocket’。
2. **连接建立：** 如果服务器支持，会返回状态码 `101 Switching Protocols`。此时，底层的 TCP 连接不会断开，而是直接复用这条通道。
3. **数据传输：** 之后双方的通信就不再遵循 HTTP 的‘请求-响应’模式了，而是通过轻量级的**二进制帧（Frame）**来互相发送数据，直到某一方主动关闭连接。

相比于传统的 HTTP 轮询（Polling），WebSocket 极大地减少了握手的开销和头部数据的冗余，非常适合像**即时通讯（IM）、实时弹幕、即时报价**这类对实时性要求极高的场景。  



在 Go 语言（比如使用 `gorilla/websocket` 库）开发时，**长连接最怕的是“假死”**（网线拔了，但服务器不知道，连接对象还挂在内存里）。

在我的 Go 项目中，为了保证连接的有效性，我实现了应用层的**心跳检测（Heartbeat）**。

服务端会定期发送 Ping 帧，客户端回复 Pong 帧，或者反过来。如果一段时间没收到心跳，服务端就会强制断开连接，释放 Goroutine 资源，防止内存泄漏。

#### [⭐️WebSocket 和 HTTP 有什么区别？](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️websocket-和-http-有什么区别)
面试官您好。

HTTP 和 WebSocket 虽然都是基于 TCP 的应用层协议，但它们在**通信模式、连接状态、数据开销以及适用场景**上有本质的区别：

**通信模式（核心区别）：**

+ **HTTP** 是**单向的请求-响应模式**（Half-Duplex）。客户端必须先发起请求，服务器才能响应。服务器无法主动向客户端推送数据。
+ **WebSocket** 是**全双工通信模式**（Full-Duplex）。一旦连接建立，客户端和服务器双方都可以随时主动向对方发送数据，不再需要‘一来一回’。

**连接状态与生命周期：**

+ **HTTP** 是**无状态、短连接**的（虽然 HTTP/1.1 有 Keep-Alive，但本质上还是为了复用 TCP 通道处理多个独立的 Request）。
+ **WebSocket** 是**有状态、长连接**的。连接一旦建立就会一直保持，直到一方主动关闭。服务器必须维护每个连接的状态（比如这个连接属于哪个用户）。

**数据开销（协议头）：**

+ **HTTP** 每次请求都要携带完整的 Header（Cookie、User-Agent 等），冗余数据多，带宽消耗大。
+ **WebSocket** 只有在握手阶段需要 HTTP Header。一旦握手完成，后续的数据传输都使用轻量级的二进制帧（Frame），头部信息极少（最小只需 2 字节），传输效率极高。

**适用场景：**

+ **HTTP** 适合传输文档、图片、API 接口等**非实时**数据。
+ **WebSocket** 适合弹幕、即时聊天（IM）、即时股价、联机游戏等对**实时性**要求极高的场景。”

#### [WebSocket 的工作过程是什么样的？](https://javaguide.cn/cs-basics/network/other-network-questions.html#websocket-的工作过程是什么样的)
WebSocket 的工作过程可以分为以下几个步骤：

1. 客户端向服务器发送一个 HTTP 请求，请求头中包含 `Upgrade: websocket`** 和 **`Sec-WebSocket-Key`** 等字段**，表示要求升级协议为 WebSocket；
2. 服务器收到这个请求后，会进行升级协议的操作，如果支持 WebSocket，它将**回复一个 HTTP 101 状态码**，响应头中包含 ，`Connection: Upgrade`和 `Sec-WebSocket-Accept: xxx` 等字段、表示成功升级到 WebSocket 协议。
3. 客户端和服务器之间建立了一个 WebSocket 连接，可以进行双向的数据传输。 数据以帧（frames）的形式进行传送，WebSocket 的每条消息可能会被切分成多个数据帧（最小单位）。 发送端会将消息切割成多个帧发送给接收端，接收端接收消息帧，并将关联的帧重新组装成完整的消息。
4. 客户端或服务器可以主动发送一个关闭帧，表示要断开连接。另一方收到后，也会回复一个关闭帧，然后双方关闭 TCP 连接。

另外，建立 WebSocket 连接之后，**通过心跳机制来保持 WebSocket 连接的稳定性和活跃性。**

#### [⭐️WebSocket 与短轮询、长轮询的区别](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️websocket-与短轮询、长轮询的区别)
面试官您好。

短轮询、长轮询和 WebSocket 都是为了解决‘服务端如何向客户端实时传递数据’的问题，但它们的实现机制和性能开销有着本质的区别：

1. **短轮询 (Short Polling)：**
    - **机制：** **客户端每隔固定时间（比如每 1 秒）向服务端发起一次完整的 HTTP 请求**。服务端无论有没有新数据，都会立刻返回响应（没有数据就返回空）。
    - **缺点：** 极其浪费网络带宽和服务器资源。大量的 HTTP 请求头部和 TCP 握手开销被白白消耗在‘无效询问’上。实时性取决于轮询的频率。
2. **长轮询 (Long Polling)：**
    - **机制：**  客户端发起 HTTP 请求后，如果服务端没有新数据，**不会立刻返回空，而是将请求‘挂起’（Hold 住连接）**。  直到有新数据产生，或者达到设定的超时时间（比如 30 秒），服务端才返回响应。客户端收到响应后，立刻发起下一次长轮询。
    - **优点与缺点：** 相比短轮询，大幅减少了无效请求的次数，降低了带宽消耗。但它依然基于 HTTP 协议，每次重新建立连接依然有 Header 开销，且服务端需要维护大量被挂起的协程/线程。
3. **WebSocket：**
    - **机制：** 客户端和服务端通过一次 HTTP 握手升级协议后，建立起一条持久的 **TCP 全双工连接**。之后双方可以随时、主动地向对方发送极其轻量级的二进制数据帧。
    - **优势：** 真正的实时通信。没有 HTTP Header 的冗余开销，延迟极低，是对服务器资源利用率最高的方案。

总结来说，短轮询是‘不断去问’，长轮询是‘去问了在那干等’，WebSocket 是‘连上了随时互通’。

#### [⭐️✨SSE 与 WebSocket 有什么区别？](https://javaguide.cn/cs-basics/network/other-network-questions.html#⭐️sse-与-websocket-有什么区别)
面试官您好。SSE (Server-Sent Events) 和 WebSocket 都是用于服务器向客户端推送数据的技术，但它们的设计初衷和适用场景有很大的区别，核心差异主要体现在四个方面：

**通信方向（最本质的区别）：**

+ **WebSocket：** 是**全双工（Bi-directional）**的。连接建立后，客户端和服务器都可以随时主动给对方发消息。就像打电话，双方都能说话。
+ **SSE：** 是**单工（Mono-directional）的。只能由服务器向客户端**发送数据，客户端不能通过 SSE 通道发消息给服务器（客户端想发数据还得发普通的 HTTP 请求）。就像收音机，你只能听，不能通过收音机回话。

**协议基础：**

+ **WebSocket：** 需要进行协议升级（Upgrade），从 HTTP 变成独立的 WebSocket 协议。
+ **SSE：** 本质上就是**标准的 HTTP 协议**。它只是把响应头的 `Content-Type` 设置为 `text/event-stream`，然后保持连接不断开，持续输出数据流。

**数据格式：**

+ **WebSocket：** 支持**文本**和**二进制**（Binary）数据，传输效率极高。
+ **SSE：** 只支持**纯文本**（UTF-8），通常传输 **JSON 字符串**。如果要传二进制文件，需要转成 Base64，效率较低。

**断线重连与特性：**

+ **WebSocket：** 协议层面没有内置重连机制，断网了需要前端写 JS 代码手动处理重连和心跳。
+ **SSE：** 浏览器原生支持**自动断线重连**，并且通过 ID 机制可以自动发送 `Last-Event-ID` header 告诉服务器从哪条消息开始补发，开发更简单。

#### 为什么 ChatGPT 的打字机效果用 SSE？
ChatGPT 的对话场景中，主要的流量是‘服务器生成内容推给用户’。

用户发出的只是短短的一句话，用普通的 HTTP POST 就够了。

而服务器回复的是长文本流。

SSE 基于 HTTP，穿透防火墙的能力比 WebSocket 强（有些企业内网会封禁非 HTTP 协议），而且 SSE 对服务端的负载更小，不用维护复杂的双向状态机。

所以在这种‘一问多答’的场景下，SSE 是性价比最高的选择。

#### 网关通常是无状态的，但 WebSocket 是有状态的长连接。当后端服务想要给特定用户推送实时信息时，它是怎么知道这个用户的 WebSocket 连接挂在哪个网关节点上的？
面试官您好

分布式的无状态网关集群中，要精准地将消息推送给特定用户的 WebSocket 长连接

核心思路是‘将长连接的状态外部化、全局化’。

主要有两种主流方案，我在设计时都有过考量：

**方案一：全局路由表模式（Registry / 集中式存储）** 这是最严谨的做法。

我会**引入 Redis 作为全局的路由注册中心**。

1. **上线绑定：** 当用户 A 连上网关节点（比如 Node-1），Node-1 会在自己的内存里保存这个 WebSocket 的句柄，同时去 Redis 里写入一条记录：`Key: UserID_A, Value: Node-1_IP`。为了防止节点宕机导致死锁，这个 Key 会设置 TTL，并由 Node-1 定期续期（心跳机制）。
2. **下发寻址：** 当后端的业务服务需要给用户 A 发消息时，它先去查 Redis，得知用户 A 连在 Node-1 上。
3. **精准投递：** 业务服务通过内部 RPC（比如 gRPC）将消息直接发送给 Node-1。Node-1 收到后，从本地内存找到对应的 WebSocket 句柄，将消息推送给用户 A。
4. **下线解绑：** 用户断开连接时，Node-1 去 Redis 删掉这条记录。

**方案二：发布订阅模式（Pub/Sub / 广播机制）** 对于实时性要求极高、且用户可能同时多端登录的场景（如弹幕推送），我会使用消息队列（如 Redis Pub/Sub 或 Kafka）。

1. **网关订阅：** 所有的网关节点都订阅一个全局的 Topic，或者按特定规则（如房间号/频道号）订阅特定的 Topic。
2. **后端发布：** 业务服务不关心用户连在哪，只管把消息发布到对应的 Topic（比如包含目标 UserID 的消息体）。
3. **网关过滤：** 所有网关节点都会收到这条消息。它们收到后，会检查自己的本地内存：‘这个目标 UserID 现在连在我这台机器上吗？’。如果在，就推送；如果不在，直接丢弃。

在实际的高并发业务（如视频网站的实时弹幕）中，考虑到弹幕大多是按‘房间/视频维度’进行群发的，我更倾向于采用 **Pub/Sub 模式**，这样业务解耦得最彻底，架构也更轻量。

#### 假设我们现在要开发一个**大型直播平台的弹幕系统**（或者 Agent 的海量实时下发控制系统）。现在有一个超级热门的直播间，单台 Go 服务器上保持着 **10 万个 WebSocket 长连接**。此时，有人发了一条弹幕，你需要把这条消息**实时广播（Broadcast）给这 10 万个在线用户**。
请问你会怎么设计这个广播机制？如果只是简单地用一个 `for` 循环遍历所有的连接去调用 `WriteMessage`，会发生什么灾难性后果？你会如何优化？

如果只是简单地用 `for` 循环遍历 10 万个连接并同步阻塞地去写数据，这绝对是灾难。

首先，单次写操作即使只要 1 毫秒，10 万次遍历完也得 100 秒，这就不是‘实时’弹幕了；

其次，一旦中间遇到几个网络极差的‘慢客户端’，底层的 TCP 发送窗口满了，`Write` 就会阻塞，直接把整个广播循环卡死，后面的用户全都被连累收不到消息。

为了解决这个问题，在 Golang 中我会采取这几层优化策略：

第一步，**引入 Channel 实现异步化**。为每个 WebSocket 连接分配一个独立的下发 Channel，广播时只负责把消息非阻塞地丢进这个 Channel，然后由连接专属的 Write Goroutine 去真正执行网络 IO。这样广播动作瞬间就能完成。

第二步，**处理慢客户端（降级与丢弃）**。既然用了 Channel，就要给它设一个合理的容量（缓冲大小）。如果在广播时发现某个客户端的 Channel 已经塞满了，说明他网络太卡消费不过来，这时候为了保护服务端内存不被撑爆，我会果断采用 `default` 分支**丢弃**这条弹幕，毕竟弹幕这种非核心数据，宁可丢弃也不能拖死整个服务。

第三步，**分片锁优化（Concurrent Map / Sharding）**。为了维护这 10 万个连接，我们肯定需要一个类似 Map 的结构。如果只用一把大锁（`sync.RWMutex`）去锁这个 Map，每次用户进出直播间都会导致严重的锁竞争。我会把这个大 Map 拆分成几十个甚至几百个小的分片（Bucket），每次增删改查只锁对应的分片，极大提高并发性能。

### 五.JWT
#### JWT 令牌和传统方式有什么区别？
+ 无状态性：
    - JWT是无状态的令牌，不需要在服务器端存储会话信息。
    - 相反，JWT令牌中包含了所有必要的信息，如用户身份、权限等。
    - 这使得JWT在分布式系统中更加适用，可以方便地进行扩展和跨域访问。
+ 安全性：
    - JWT使用密钥对令牌进行签名，确保令牌的完整性和真实性。
    - 只有持有正确密钥的服务器才能对令牌进行验证和解析。
    - 这种方式比传统的基于会话和Cookie的验证更加安全，有效防止了CSRF（跨站请求伪造）等攻击。
+ 跨域支持：
    - JWT令牌可以在不同域之间传递，适用于跨域访问的场景。
    - 通过在请求的头部或参数中携带JWT令牌，可以实现无需Cookie的跨域身份验证。

#### **了解过你们项目的鉴权机制吗？为什么不用传统的 Session？**
因为我们是微服务架构，并且业务可能有高并发的需求。

使用 Session 就涉及到服务端状态同步的问题（Session 集中存储的单点瓶颈）。

使用 Token 机制（如 PASETO）则是无状态的，Api 网关甚至任意微服务节点都可以独立通过密钥完成对 Token 的自验证，不需要每次向中心服务器去做 DB 查询比对，扩展性极强。

#### ✨JWT 怎么踢人下线？
既然 JWT 签发后服务端就不管了，那如果发现用户账号被盗，或者你在后台封禁了一个用户，**怎么让他手里的 JWT 立刻失效？**

面试官您好。

JWT 本质是**无状态的**，**一旦签发在过期前自身是无法作废的。**

但在实际业务中，为了满足‘踢人下线’、‘单点登录（互踢）’或‘账号封禁’的需求，我们通常会引入轻量级的状态管理

主要有以下三种解决方案：

1. **方案一：引入 Redis 黑名单机制（最常用）**  当我们需要踢某个用户下线时，将该用户当前 JWT 的唯一标识（`jti` 字段）或者完整的 Token 存入 Redis，并设置过期时间（TTL）与该 JWT 的剩余有效期一致。 网关层每次在校验 JWT 签名合法后，还会去 Redis 查一下这个 Token 是否在黑名单中。 如果在，直接返回 401 拒绝访问。
2. **方案二：基于版本号机制 (Token Versioning)**  在数据库的用户表中增加一个 `token_version`（整数）字段。 每次用户登录签发 JWT 时，把这个版本号写进 JWT 的 payload 里。 当需要踢人下线或修改密码时，将数据库中的 `token_version` 加 1。 用户带旧 Token 请求时，拦截器会对比 JWT 里的版本号和数据库/缓存里的版本号，不一致则判定失效。
3. **方案三：双 Token 缩短危险窗口期 (Access + Refresh Token)**  不直接暴力踢人，而是将负责业务请求的 `Access Token` 有效期设置得极短（比如 5-15 分钟），而将用来刷新令牌的 `Refresh Token` 存在服务端的数据库或 Redis 中。 踢人下线时，直接作废服务端的 `Refresh Token`。用户最多在几分钟后 Access Token 过期，就再也无法获取新 Token，自然就被踢下线了。

在实际的高并发微服务架构中，为了兼顾性能和安全性，我们通常会采用**短有效期的 Access Token + Redis 黑名单**的组合方案。

#### 现在业界 90% 的项目都在用 JWT，你为什么在网关里选择了 PASETO？它解决了 JWT 的什么痛点？
面试官您好，我在项目中之所以在网关层采用 PASETO 而非传统的 JWT，主要是出于安全性（Secure by Default）的考量。

虽然 JWT 极其流行，但它在设计上把过多的选择权交给了开发者，容易导致严重的安全漏洞。

比如：

1. **算法混淆攻击（Algorithm Confusion）：**  
    1. JWT 的 Header 里有一个 `alg` 字段用来指定加密算法。 
    2. 黑客可以恶意将非对称加密算法（如 RS256）修改为对称加密算法（如 HS256），然后用服务端的公钥作为对称密钥去伪造签名，导致服务端验证通过。
2. **"None" 算法漏洞：** 
    1. JWT 规范甚至允许 `alg: none`，如果不小心在代码里没校验好，黑客就能直接绕过签名。

而 **PASETO (Platform-Agnostic Security Tokens)** 的核心理念是‘默认安全’。

 它摒弃了这种让开发者自由拼凑加密算法的做法，而是采用了**版本化（Version）和用途隔离（Local/Public）**的设计。

比如我使用的是 `v2.local`（对称加密)，它底层直接写死了目前最安全的加密套件（如 Ed25519）。

这从协议层面上彻底杜绝了算法篡改和降级攻击，降低了我们在网关开发时踩坑的安全风险。

#### ✨PASETO 和JWT的区别是什么？
面试官您好。我在实现 go-zero 网关鉴权时，深入对比过 JWT 和 PASETO，它们的核心区别主要体现在**设计哲学、算法协商机制以及载荷（Payload）的机密性**上：

1. **设计哲学不同（灵活度 vs 默认安全）：** 
    1. **JWT** 的设计理念是‘密码学敏捷（Cryptographic Agility）’，它给了开发者极大的自由去选择各种加密算法，但这往往导致配置错误和安全漏洞。
    2. **PASETO** 的核心理念是‘默认安全（Secure by Default）’。它刻意剥夺了开发者的选择权，将最优的加密套件强行绑定在特定的‘版本’中，从根源上杜绝了错误使用的可能。
2. **头部与算法机制（核心痛点）：**
    - **JWT** 的 Header 中包含一个 `alg`（算法）字段，这意味着 Token 在一定程度上可以“告诉”服务端该用什么算法来解密它。这导致了 JWT 历史上最危险的两个漏洞：**None 算法漏洞**（黑客把 alg 改为 none 绕过签名校验）和**算法混淆攻击**（把非对称的 RSA 改为对称的 HMAC，用公钥伪造签名）。
    - **PASETO** 彻底废弃了 `alg` 字段。它采用的是 `版本号.场景.载荷.页脚` 的结构。比如 `v4.local` 或 `v4.public`。只要你选定了版本和场景，底层的加密算法（如 Ed25519 或 XChaCha20）就是唯一且写死的，黑客根本无从篡改。
3. **载荷的机密性（加密 vs 明文）：**
    - 我们平时常用的 **JWT**（准确说是 JWS）只是对数据进行了 Base64 编码和防篡改签名，**Payload 是明文的**。任何人拿到 Token 都能轻易解码出里面的 UserID 或角色信息。
    - 而在 **PASETO** 中，如果使用 `local`（对称加密）模式，它不仅会签名防篡改，还会默认使用 AEAD（认证加密）将 **Payload 完全加密成密文**。即使 Token 被抓包，别人也无法窥探到内部的任何业务数据。

综上所述，PASETO 通过强约束和默认加密，极大地降低了我们在网关开发时踩坑的安全风险。

#### JWT 令牌都有哪些字段？
JWT 本质上就是一组字串，通过（`.`）切分成三个为 Base64 编码的部分：

+ **Header（头部）** : 
    - 描述 JWT 的元数据，定义了**生成签名的算法以及 **`**Token**`** 的类型**。
    - Header 被 Base64Url 编码后成为 JWT 的第一部分。
+ **Payload（载荷）** : 
    - 用来**存放实际需要传递的数据**，包含声明（Claims），如`sub`（subject，主题）、`jti`（JWT ID）。
    - Payload 被 Base64Url 编码后成为 JWT 的第二部分。
+ **Signature（签名）**：
    - 服务器通过 Payload、Header 和一个密钥(Secret)使用 Header 里面指定的签名算法（默认是 HMAC SHA256）生成。
    - 生成的签名会成为 JWT 的第三部分。

JWT 通常是这样的：`xxxxx.yyyyy.zzzzz`。

#### [如何基于 JWT 进行身份验证？](https://javaguide.cn/system-design/security/jwt-intro.html#如何基于-jwt-进行身份验证)
在基于 JWT 进行身份验证的应用程序中，服务器通过 Payload、Header 和 Secret(密钥)创建 JWT 并将 JWT 发送给客户端。

客户端接收到 JWT 之后，会将其保存在** Cookie 或者 localStorage **里面，以后**客户端发出的所有请求都会携带这个令牌。**

1. 用户向服务器发送**用户名、密码以及验证码**用于登陆系统；
2. 如果用户**用户名、密码以及验证码校验正确**的话，**服务端会返回已经签名的 Token，也就是 JWT；**
3. 客户端收到 Token 后自己保存起来（比如浏览器的 `localStorage` ）；
4. 用户以后每次向后端发请求都**在 Header 中带上这个 JWT** ；
5. 服务端检查 JWT 并从中获取用户相关信息。

#### [⭐️什么时候选择 TCP，什么时候选 UDP?](https://javaguide.cn/cs-basics/network/other-network-questions2.html#⭐️什么时候选择-tcp-什么时候选-udp)
选择 TCP 还是 UDP，主要取决于你的应用**对数据传输的可靠性要求有多高，以及对实时性和效率的要求有多高**。

当**数据准确性和完整性至关重要，一点都不能出错**时，通常选择 TCP。因为 TCP 提供了一整套机制（三次握手、确认应答、重传、流量控制等）来保证数据能够可靠、有序地送达。典型应用场景如下：

+ **Web 浏览 (HTTP/HTTPS):** 网页内容、图片、脚本必须完整加载才能正确显示。
+ **文件传输 (FTP, SCP):** 文件内容不允许有任何字节丢失或错序。
+ **邮件收发 (SMTP, POP3, IMAP):** 邮件内容需要完整无误地送达。
+ **远程登录 (SSH, Telnet):** 命令和响应需要准确传输。

在实际用 Go 语言开发基于 TCP 的服务端时，因为 TCP 是字节流，客户端连续发送两个 100 字节的包，服务端在 **conn.Read** 时可能一次性读出了一个 200 字节的‘粘包’，也可能读出了 50 字节的‘半包’。

所以我们通常需要在应用层自己定义通信协议，比如使用 **Length-Field Based Frame Decoder（基于长度字段的帧解码）**，在消息头部固定几个字节用来标明消息体的长度，从而准确地切分每个完整的数据包。

当**实时性、速度和效率优先，并且应用能容忍少量数据丢失或乱序**时，通常选择 UDP。UDP 开销小、传输快，没有建立连接和保证可靠性的复杂过程。典型应用场景如下：

+ **实时音视频通信 (VoIP, 视频会议, 直播):** 偶尔丢失一两个数据包（可能导致画面或声音短暂卡顿）通常比因为等待重传（TCP 机制）导致长时间延迟更可接受。应用层可能会有自己的补偿机制。
+ **在线游戏:** 需要快速传输玩家位置、状态等信息，对实时性要求极高，旧的数据很快就没用了，丢失少量数据影响通常不大。
+ **DHCP (动态主机配置协议):** 客户端在请求 IP 时自身没有 IP 地址，无法满足 TCP 建立连接的前提条件，并且 DHCP 有广播需求、交互模式简单以及自带可靠性机制。
+ **物联网 (IoT) 数据上报:** 某些场景下，传感器定期上报数据，丢失个别数据点可能不影响整体趋势分析。

事实上，TCP 由于其底层的机制，存在严重的‘队头阻塞’问题，并且三次握手的延迟在现代移动互联网下显得太慢了。

目前业界最前沿的趋势是**基于 UDP 来实现可靠传输**。

比如 Google 主导的 **QUIC 协议**（也是 HTTP/3 的底层基础）。

它在底层使用了速度极快的 UDP，而在**应用层用代码自己实现了类似 TCP 的可靠性、流量控制和加密（集成了 TLS）**。这证明了 UDP 的轻量级是未来突破网络性能瓶颈的关键。

#### TCP 粘包和拆包到底怎么用代码解决
TCP 之所以会粘包拆包，是因为它是一个“**面向无边界的字节流**”协议。

在 TCP 眼里，没有“一条消息”的概念，只有一堆连绵不断的水流。

面试官您好。

TCP 粘包和拆包是因为 TCP 是面向字节流的，没有消息边界。

要在代码层面解决这个问题，核心思想是**在应用层定义消息边界**。业界主要有三种通用解决方案：

1. **定长消息法 (Fixed Length)：**
    1.  规定每条消息必须是**固定的长度**（比如 100 字节）。如果有效数据不够，就用空格或 `\0` 补齐。接收端每次死死地读满 100 字节就算收到一条完整消息。
    2. **缺点是极其浪费带宽，实战中极少使用。**
2. **特殊分隔符法 (Delimiter Based)：** 
    1. 在每条完整消息的末尾追加一个特殊字符（比如 HTTP 协议里的 `\r\n`）。
    2. 接收端持续读取字节流，直到扫描到这个分隔符，就把前面读到的数据当做一条完整的消息。
    3. **缺点是如果消息体本身就包含了这个分隔符，需要做复杂的转义处理。**
3. **长度字段法 / TLV 格式 (Length-Field Based) —— 最主流推荐：** 这也是绝大多数 RPC 框架底层采用的方案。
    1. 我们将一条消息分为 `Header` 和 `Body`。
    2. 在固定长度的 Header 中，存入一个数字，用来表示接下来的 Body 有多长。 
    3. 接收端的工作分为两步：
        * 先读取固定长度的 Header，解析出 Body 的长度 `N`。
        * 接着再从字节流里往后严格读取 `N` 个字节的数据。这样就完美解决了粘包和拆包的问题。

#### TCP 的三次握手和四次挥手全过程（包含状态流转）
面试官您好。TCP 作为一个可靠的、面向连接的协议，其建立连接和断开连接的过程就是经典的‘三次握手’和‘四次挥手’。

**第一部分：三次握手（建立连接）** 起初，**服务端处于 **`**LISTEN**`**（监听）状态，等待客户端的连接。**

1. **第一次握手：** 
    1. 客户端向服务端发送一个带有 `SYN = 1` 标志位的数据包，并随机生成一个初始序列号 `seq = x`。
    2. 此时，客户端进入 `SYN_SENT` 状态。
2. **第二次握手：** 
    1. 服务端收到后，需要确认客户端的报文，于是发送一个带有 `SYN = 1` 和 `ACK = 1` 的数据包，同时指定自己的初始序列号 `seq = y`，并确认号 `ack = x + 1`。
    2. 此时，服务端进入 `SYN_RCVD`（同步收到）状态。
3. **第三次握手：** 
    1. 客户端收到服务端的确认后，再向服务端发送一个带有 `ACK = 1` 的数据包，确认号 `ack = y + 1`，序列号 `seq = x + 1`。
    2. 发送完毕后，客户端进入 `ESTABLISHED`（已建立连接）状态。
    3. 服务端收到这个包后，也进入 `ESTABLISHED` 状态。至此，全双工连接建立。



**第二部分：四次挥手（断开连接）** 假设**客户端主动发起断开连接**。

1. **第一次挥手：** 
    1. 客户端发送一个 `FIN = 1` 的数据包（代表客户端没有数据要发送了），序列号 `seq = u`。
    2. 此时客户端进入 `FIN_WAIT_1` 状态。
2. **第二次挥手：** 
    1. 服务端收到 FIN 包后，立刻回复一个 `ACK = 1` 的确认包，确认号 `ack = u + 1`。
    2. 此时服务端进入 `CLOSE_WAIT` 状态。
    3. 客户端收到确认后，进入 `FIN_WAIT_2` 状态。（此时连接处于“半关闭”状态，客户端不能发数据了，但服务端如果还有数据没发完，还可以继续发）。
3. **第三次挥手：** 
    1. 等服务端的数据全部发送完毕，它也会向客户端发送一个 `FIN = 1` 的数据包，序列号 `seq = w`。
    2. 此时服务端进入 `LAST_ACK`（最后确认）状态。
4. **第四次挥手：** 
    1. 客户端收到服务端的 FIN 包后，回复一个 `ACK = 1` 的包，确认号 `ack = w + 1`。
    2. 此时客户端进入著名的 `TIME_WAIT` 状态。
    3. 服务端收到确认后，立刻进入 `CLOSED` 状态。
    4. 而客户端要在 `TIME_WAIT` 状态等待 **2MSL**（最大报文段生存时间）后，才最终进入 `CLOSED` 状态。

#### 为什么是“三”次握手？两次不行吗？四次呢？
**为什么不能是两次？** 最核心的原因是**为了防止“历史的重复连接初始化”造成混乱**。

假设客户端发出的第一个 SYN 包在网络中滞留了，客户端超时后又发了一个新的 SYN。

如果只有两次握手，服务端收到那个旧的滞留 SYN，立刻就单方面建立连接并分配资源了，这就造成了资源的浪费和状态错误。

三次握手允许客户端在第三步有权去“拒绝”那个由于历史延迟而产生的错误连接（发一个 RST 包中止它）。另外，三次握手也是为了让双方互相确认对方的“接收”和“发送”能力都是正常的。

**为什么不需要四次？** 

第二次握手实际上把服务端的 `ACK`（确认收到客户端请求）和 `SYN`（服务端发起自己的同步请求）合并成了一个包发送，省去了一次网络交互，所以不需要四次。

#### 为什么挥手需要“四”次？
这要扣住 TCP 的“全双工（Full-Duplex）”_特性。 _

_三次握手时，_`SYN`_ 和 _`ACK`_ 可以合并发。_

_但是挥手时，当服务端收到客户端的 _`FIN`_，只能说明客户端不发数据了_，但**服务端可能还有数据没处理完/没发完**。

所以服务端只能先回一个 `ACK`（告诉客户端我收到了），等自己真把活儿干完了，再单独发一个自己的 `FIN`（告诉客户端我也发完了）。这就是为什么挥手比握手多了一次，中间那个 `CLOSE_WAIT` 状态就是用来把剩余数据发完的。

+ **<font style="color:rgb(60, 60, 67);">顶级域 DNS 服务器（TLD 服务器）。</font>**<font style="color:rgb(60, 60, 67);">顶级域是指域名的后缀，如</font>`<font style="color:rgb(60, 60, 67);">com</font>`<font style="color:rgb(60, 60, 67);">、</font>`<font style="color:rgb(60, 60, 67);">org</font>`<font style="color:rgb(60, 60, 67);">、</font>`<font style="color:rgb(60, 60, 67);">net</font>`<font style="color:rgb(60, 60, 67);">和</font>`<font style="color:rgb(60, 60, 67);">edu</font>`<font style="color:rgb(60, 60, 67);">等。国家也有自己的顶级域，如</font>`<font style="color:rgb(60, 60, 67);">uk</font>`<font style="color:rgb(60, 60, 67);">、</font>`<font style="color:rgb(60, 60, 67);">fr</font>`<font style="color:rgb(60, 60, 67);">和</font>`<font style="color:rgb(60, 60, 67);">ca</font>`<font style="color:rgb(60, 60, 67);">。TLD 服务器提供了权威 DNS 服务器的 IP 地址。</font>
+ **<font style="color:rgb(60, 60, 67);">权威 DNS 服务器</font>**<font style="color:rgb(60, 60, 67);">。在因特网上具有公共可访问主机的每个组织机构必须提供公共可访问的 DNS 记录，这些记录将这些主机的名字映射为 IP 地址。</font>
+ **<font style="color:rgb(60, 60, 67);">本地 DNS 服务器</font>**<font style="color:rgb(60, 60, 67);">。每个 ISP（互联网服务提供商）都有一个自己的本地 DNS 服务器。当主机发出 DNS 请求时，该请求被发往本地 DNS 服务器，它起着代理的作用，并将该请求转发到 DNS 层次结构中。严格说来，不属于 DNS 层级结构</font>

<font style="color:rgb(60, 60, 67);">世界上并不是只有 13 台根服务器，这是很多人普遍的误解，网上很多文章也是这么写的。实际上，现在根服务器数量远远超过这个数量。最初确实是为 DNS 根服务器分配了 13 个 IP 地址，每个 IP 地址对应一个不同的根 DNS 服务器。然而，由于互联网的快速发展和增长，这个原始的架构变得不太适应当前的需求。为了提高 DNS 的可靠性、安全性和性能，目前这 13 个 IP 地址中的每一个都有多个服务器，截止到 2023 年底，所有根服务器之和达到了 1700 多台，未来还会继续增加。</font>

#### [⭐️DNS 解析的过程是什么样的？](https://javaguide.cn/cs-basics/network/other-network-questions.html#%E2%AD%90%EF%B8%8Fdns-%E8%A7%A3%E6%9E%90%E7%9A%84%E8%BF%87%E7%A8%8B%E6%98%AF%E4%BB%80%E4%B9%88%E6%A0%B7%E7%9A%84)
简单来说，DNS 解析就是把人类能看懂的域名（比如 `www.google.com`）翻译成计算机能看懂的 IP 地址的过程。

整个过程像是一个**层层递进的问路过程**：

1. **查本地通讯录（本地缓存）**：
    1. **浏览器首先会看看自己的缓存里有没有这个域名的 IP**；
    2. 如果没有，就去问操作系统的缓存（包括 `hosts` 文件）；
    3. 如果还没有，再去问家里的路由器缓存。
2. **求助本地查号台（Local DNS）**：
    1. 如果本地都没找到，操作系统就会向你配置的**本地 DNS 服务器**（比如运营商提供的，或者像 `8.8.8.8`、`114.114.114.114`）发起请求。
    2. 这个请求是**递归**的，意思是：‘我不管你怎么找，你必须给我一个最终结果’。
3. **问总机（根域名服务器 Root）**：
    1. Local DNS 发现自己也不知道，它就会去问全球级别的‘根域名服务器’。
    2. 根服务器说：‘我不知道具体的，但我知道 `.com` 的负责人是谁，你去问它’。
4. **问分机（顶级域名服务器 TLD）**：
    1. Local DNS 接着去问 `.com` 的顶级域名服务器。
    2. 顶级服务器说：‘我也不知道具体的 IP，但我查到了 `google.com` 这个域名的权威服务器在哪，你去问它’。
5. **找最终负责人（权威域名服务器 Authoritative）**：
    1. Local DNS 最终向 `google.com` 的权威服务器发起询问。
    2. 权威服务器查了查自己的记录，说：‘找到了，它的 IP 是 `192.168.x.x`’。
6. **记录并返回**：最后，Local DNS 把这个宝贵的 IP 地址存到自己的缓存里，然后把它交还给操作系统和浏览器。寻路结束，接下来就可以发起 TCP 握手了。

#### [DNS 劫持了解吗？如何应对？](https://javaguide.cn/cs-basics/network/other-network-questions.html#dns-%E5%8A%AB%E6%8C%81%E4%BA%86%E8%A7%A3%E5%90%97-%E5%A6%82%E4%BD%95%E5%BA%94%E5%AF%B9)
<font style="color:rgb(60, 60, 67);">DNS 劫持是一种网络攻击，它通过</font>**<font style="color:rgb(60, 60, 67);">修改 DNS 服务器的解析结果</font>**<font style="color:rgb(60, 60, 67);">，</font>**<font style="color:rgb(60, 60, 67);">使用户访问的域名指向错误的 IP 地址</font>**<font style="color:rgb(60, 60, 67);">，从而导致用户无法访问正常的网站，或者被引导到恶意的网站。</font>

<font style="color:rgb(60, 60, 67);">DNS 劫持有时也被称为 DNS 重定向、DNS 欺骗或 DNS 污染。</font>
