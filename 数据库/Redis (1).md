## 一.数据结构
### Redis 有什么作用?为什么要用 Redis/为什么要用缓存？  
 Redis 本质上是**内存 KV 数据库**（支持多种数据结构）,常见用法有这些：

1. **做缓存**：把热点数据放内存里（商品详情、用户信息、配置等），下次直接拿，不用每次查 MySQL。
2. **做分布式锁**：控制并发，防止超卖/重复下单/重复执行任务。
3. **做验证码/临时数据**：短信码、邮箱码、登录失败次数、黑名单 token 这种“过一会就没用”的数据（带过期时间）。
4. **计数&限流**：访问次数统计、接口限流（比如 1 分钟最多 60 次）、点赞/阅读量。
5. **排行榜**：积分榜、热度榜（Redis 的 ZSET 很适合）。
6. **轻量消息队列**：异步任务、削峰（Stream/List，轻量可用）。

 更快 + 更抗压 + 更省数据库成本  

+ **更快（降低延迟）**  
Redis 在内存里，读写非常快；MySQL 查一次通常更慢。接口响应时间会明显下降。
+ **更抗压（保护数据库）**  
流量一大（热点/大促/秒杀），如果都打 MySQL 容易把数据库打崩。  
有缓存后，大部分请求被 Redis 接住，数据库压力小很多。
+ **更省事/更容易扩展**  
没缓存可能很快就要分库分表、加很多 DB 机器。  
先用缓存把读流量挡住，系统更容易撑住增长。

 用缓存会有坑，但有办法  

+ **穿透**：一直查不存在的数据 → 打爆 DB（缓存空值/布隆过滤器）
+ **击穿**：热点 key 过期瞬间很多人回源 → 打爆 DB（加锁/单飞 singleflight）
+ **雪崩**：大量 key 同时过期或 Redis 故障 → 流量全打 DB（TTL 加随机、限流降级）
+ **一致性**：DB 更新了缓存还是旧的 → 常见做法“**先写 DB，再删缓存**”。

### <font style="color:rgb(51, 51, 51);">singleflight 怎么用的，具体讲讲原理（pcg）</font>
面试官您好，关于 `singleflight`，它是 Go 官方扩展同步包（`x/sync`）里一个非常强大的并发控制工具。

在我的系统里，我主要用它来**解决缓存击穿（Cache Breakdown）**的问题。

**第一：它解决了什么业务痛点？** 

假设我的金融资讯平台有一篇爆款新闻，瞬间涌入 1 万个并发读请求，而此时 Redis 里的缓存刚好过期失效。

如果没有保护，这 1 万个请求会全部打穿到 MySQL，数据库瞬间就会宕机。 

引入 `singleflight` 后，它能在内存层面把这 1 万个相同 Key 的请求**合并为 1 个**。

最终只有第 1 个请求去查数据库，剩下的 9999 个请求在内存里阻塞等待，等第 1 个请求拿到数据后，大家**共享结果**直接返回。

**第二：代码层面是怎么用的？** 

使用起来非常简洁。

我会初始化一个全局的 `singleflight.Group`，然后调用它的 `Do(key, fn)` 方法。

`key` 通常是业务的缓存键（比如文章 ID），`fn` 则是真正去查数据库的闭包函数。

实际上，在我使用的 **Go-Zero 框架**中，底层缓存组件已经深度封装了这个机制（在源码里叫 `SharedCalls`），我只要按规范写查库逻辑，框架天然就具备了防击穿能力。

**第三：它的底层原理是什么？（重点大招）** 

如果扒开它的源码，它的底层核心其实就是三样东西：`**Map**`** + **`**Mutex**`** + **`**WaitGroup**`。 

在 `Group` 结构体里，它维护了一个 `map[string]*call`，用来记录当前正在执行的请求。

1. 当第 1 个请求进来时，加互斥锁，发现 Map 里没自己，就会创建一个 `call` 对象塞进 Map，并调用 `WaitGroup.Add(1)`，然后放开锁，自己去慢吞吞地查数据库。
2. 当后面 9999 个请求进来时，一查 Map 发现这个 key 已经有人在处理了，就会直接调用那个 `call` 对象的 `**WaitGroup.Wait()**`，把自己阻塞挂起。
3. 等第 1 个请求查完数据库，把结果赋值给 `call` 对象，然后执行 `**WaitGroup.Done()**`，并把 key 从 Map 里删掉。
4. 此时，阻塞等待的 9999 个协程瞬间被唤醒，直接拿走内存里的结果返回。整个过程极其优雅地保护了下游存储。”





Singleflight 的核心作用是：请求合并（Request Coalescing）。  

当有 10,000 个并发请求同时去查同一个 `Key`（比如“热门文章ID: 10086”），而这个 Key 刚好在 Redis 里过期了。

+ **不用 Singleflight：** 10,000 个请求全部打穿到 MySQL，数据库瞬间 CPU 100% 宕机。
+ **用了 Singleflight：** 这 10,000 个请求在内存里“排队”。系统只派**第 1 个**请求去 MySQL 查数据。等它查回来，把结果**共享**给剩下等待的 9,999 个请求。数据库只收到 1 次查询。

 Singleflight 的原理非常精妙，总结起来就是：**Map + Mutex + WaitGroup**。  



Singleflight 就是大家排队，派一个人去买饭，回来分给大家吃。底层是用 Map 记录谁去买饭了，用 WaitGroup 让其他人等着。  

 

在使用上，它极其简单，核心方法只有两个：`Do` 和 `DoChan`。  

```go
package main

import (
	"fmt"
	"sync"
	"time"
	"golang.org/x/sync/singleflight" // 需要引入这个官方扩展包
)

// 定义一个全局的 group
var g singleflight.Group

func GetArticleDetail(articleID int) (string, error) {
	key := fmt.Sprintf("article:%d", articleID)

	// 核心代码：g.Do
	// 参数1：key (同一个 key 的请求会被合并)
	// 参数2：fn (真正去执行查询的函数)
	// 返回值 v: 查询结果
	// 返回值 err: 错误信息
	// 返回值 shared: bool，表示结果是否被多个请求共享了
	v, err, shared := g.Do(key, func() (interface{}, error) {
		// 模拟查数据库的耗时
		fmt.Println("--> 正在去数据库查询数据...", key) 
		time.Sleep(1000 * time.Millisecond)
		return "文章内容: 震惊！某后端程序员...", nil
	})

	if err != nil {
		return "", err
	}

	fmt.Printf("请求结束，结果: %v, 是否共享: %v\n", v, shared)
	return v.(string), nil
}

func main() {
	var wg sync.WaitGroup
	// 模拟 10 个并发请求同时进来
	for i := 0; i < 10; i++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			GetArticleDetail(10086)
		}()
	}
	wg.Wait()
}
```

### Singleflight 能解决分布式系统的缓存击穿吗？  
+ **答案：****不能完全解决，只能减轻。**
+ **原因：** 
    - Singleflight 是**单机**的（In-Memory）。
    - 如果你部署了 10 个微服务节点，每个节点会有一个请求去查 DB，DB 依然会收到 10 个请求（而不是 10,000 个）。
+ **补救：** 对于 10 个请求也不想查的情况，需要引入**分布式锁**（Redis SetNX）。但在实际工程中，10 个并发请求对 DB 通常是可以接受的，所以单机 Singleflight 已经足够优秀了。

### <font style="color:rgb(51, 51, 51);">为什么要用 redis 做限流（pcg）</font>
**打破单机孤岛，实现全局精准控流  **

我们的 API 网关为了高可用，通常是多节点部署的（比如 10 台 Pod）。

如果使用本地内存限流（比如给 `UserA` 限制 100次/秒），由于负载均衡（Nginx/K8s Ingress）会将请求打散，`UserA` 实际上可以在整个集群里发出 10*100 = 1000次请求，**本地限流在分布式环境下直接失效了**。 

而 Redis 作为一个独立于网关之外的**全局中心化存储**，所有网关节点都去向它请示。

**它能保证业务上（比如限制某个 IP 或某个 UserID）绝对的全局精准限流。**

****

**数据结构天然契合限流算法  **

Redis 的数据结构简直是为了限流而生的：

+ 固定窗口算法： 
    - 直接用最基础的 `String` 加 `INCR` 和 `EXPIRE` 指令，三行代码搞定。
+ 滑动窗口算法： 
    - 完美契合 `ZSet`。
    - 把请求的时间戳作为 Score，利用 `ZCOUNT` 统计当前时间窗口内的请求数，再用 `ZREMRANGEBYSCORE` 剔除过期数据，逻辑极其优雅。

****

**基于单线程模型与 Lua 脚本的“绝对原子性”  **

限流的核心逻辑是‘判断当前余量 -> 扣减余量’，这是一个典型的并发竞态条件（Race Condition）。 

如果用传统的数据库，还要加分布式锁，极度消耗性能。

而 Redis 是**单线程处理命令**的，我们可以把令牌桶或漏桶算法的逻辑写成一段 **Lua 脚本**发给 Redis 执行。

Redis 保证 Lua 脚本的执行是**绝对原子性**的，不需要任何外部锁，完美解决了高并发下的限流超卖问题。  



**极致的内存读写性能  **

限流是每一个 HTTP 请求进网关时都要经历的‘前置拦截器’。

这就要求限流组件的延迟必须在 1~2 毫秒以内。Redis 基于内存操作，单节点轻松扛住 10 万+ QPS，完全能跟得上网关的超高吞吐量，不会成为整个请求链路的性能瓶颈。

### Redis 常用的数据结构有哪些？
总结：

+ **String** 缓存/计数
+ **Hash** 存对象字段
+ **List** 队列
+ **Set** 去重/集合运算
+ **ZSet** 排行榜
+ **Bitmap/HyperLogLog** 统计类省内存
+ **Geo** 附近
+ **Stream** 消息队列



1）String（字符串）

+ **能干啥**：最基础的 KV；也能做计数
+ **场景**：缓存对象 JSON、token、验证码、计数器
+ **常用命令**：`GET/SET`、`SETEX`、`INCR/DECR`

2）Hash（哈希/字典）

+ **能干啥**：一个 key 下面放很多 field（像一行数据）
+ **场景**：存用户信息（name/age/phone）、商品属性
+ **常用命令**：`HSET/HGET/HMGET`、`HGETALL`

3）List（列表）

+ **能干啥**：有序链表，两端 push/pop
+ **场景**：简单队列、消息缓冲、最新 N 条记录（也能用）
+ **常用命令**：`LPUSH/RPUSH`、`LPOP/RPOP`、`BRPOP`

4）Set（集合）

+ **能干啥**：无序、不重复
+ **场景**：去重（比如 UV）、共同好友（交集）、抽奖名单
+ **常用命令**：`SADD/SREM`、`SISMEMBER`、`SMEMBERS`、`SINTER/SUNION`

5）ZSet（有序集合）

+ **能干啥**：Set + 分数 score，按 score 排序
+ **场景**：排行榜、TopN、按时间/热度排序列表
+ **常用命令**：`ZADD`、`ZRANGE/ZREVRANGE`、`ZRANK`、`ZINCRBY`

6）Bitmap（位图）

+ **能干啥**：用 bit 表示状态，超省空间
+ **场景**：签到（第 N 天是否签到）、用户是否在线、布尔状态统计
+ **常用命令**：`SETBIT/GETBIT`、`BITCOUNT`

7）HyperLogLog

+ **能干啥**：近似去重计数（省内存，但不是精确值）
+ **场景**：UV 统计、独立访客数
+ **常用命令**：`PFADD`、`PFCOUNT`、`PFMERGE`



8）Geo（地理位置）

+ **能干啥**：经纬度存储、距离/范围查询
+ **场景**：附近的人、附近的店
+ **常用命令**：`GEOADD`、`GEODIST`、`GEOSEARCH`

9）Stream（消息流）

+ **能干啥**：更像 MQ：消费组、ACK、pending
+ **场景**：异步任务、事件流、轻量消息队列
+ **常用命令**：`XADD`、`XREAD`、`XREADGROUP`、`XACK`

### 介绍一下Bitmap（字节）
Bitmap 本质上是一个连续的、由 bit（比特位）组成的数组。

在计算机中，最小的寻址单位是字节（Byte，1 Byte = 8 bits），而 Bitmap 打破了这个限制，直接在 bit 级别上操作数据。它通常利用 `0` 和 `1` 两个状态，来映射某个 ID 对应的业务状态（例如：是否在线、是否签到、是否点过赞）。  

+ **极致的空间利用率：** 传统的整型（如 Go 中的 `int32`）需要占用 4 个字节（32 bits）来存储一个状态。而 Bitmap 只需要 1 个 bit。这意味着 1 亿个用户的某种布尔状态（比如今天是否登录），使用常规的 `int32` 数组大概需要 400 MB 内存，而使用 Bitmap 仅仅需要大约 12 MB。
+ **极高的运算效率：** 计算机处理位运算（交集 `&`、并集 `|`、异或 `^`）的速度远远快于普通的算术运算。这使得在处理多天的状态统计时，性能表现极佳。

在微服务和缓存设计中，Bitmap 的出场率非常高，尤其是在 Redis 中（底层基于字符串类型实现）。

+ **海量用户签到统计：** 按年为单位，一年 365 天对应 365 个 bit。使用 Redis 的 `SETBIT key offset value` 命令，通过计算二进制中 `1` 的个数（`BITCOUNT`）就能快速得出总签到天数。
+ **活跃用户（DAU/MAU）统计：** 每天使用一个 Bitmap，将用户的 ID 作为 offset（偏移量）。如果当天登录过，就将对应 bit 置为 1。计算连续一周活跃的用户，只需要把这 7 天的 Bitmap 做按位与（AND）运算即可。
+ **布隆过滤器（Bloom Filter）的底层基石：** 在处理高并发的缓存穿透问题时，布隆过滤器的底层就是由一个超大的 Bitmap 和多个 Hash 函数组成的。通过 Hash 计算出多个 bit 位置为 1，来实现以极小的内存判断“数据绝对不存在或可能存在。



### 讲一下Redis底层的数据结构
$ \text{Redis} $ 是一个键值（Key-Value）存储系统，它使用一个**全局的哈希表**来存储所有的键值对。

但关键在于，这个哈希表中的 $ \text{Value} $** 可以是五种基本数据类型**，而每种数据类型在底层又会根据**存储的数据量和数据大小**选择使用不同的**底层数据结构**，以**优化内存和性能**。

Redis常见有五种数据结构：

| **Redis 数据类型** | **底层数据结构 (小数据量/精简模式)** | **底层数据结构 (大数据量/标准模式)** |
| --- | --- | --- |
| **String** (字符串) | $ \text{Int} $ (整数) | $ \text{SDS} $ (Simple Dynamic String) |
| **List** (列表) | $ \text{ZipList} $ (压缩列表) | $ \text{LinkedList} $ (双向链表) 或 $ \text{QuickList} $ (快速列表) |
| **Hash** (哈希) | $ \text{ZipList} $ (压缩列表) | $ \text{HashTable} $ (哈希表) |
| **Set** (集合) | $ \text{IntSet} $ (整数集合) | $ \text{HashTable} $ (哈希表) |
| **ZSet** (有序集合) | $ \text{ZipList} $ (压缩列表) | $ \text{Skip List} $ (跳表) + $ \text{HashTable} $ |


**String（字符串），Hash（哈希），List（列表），Set（集合）、Zset（有序集合）**。

随着版本的更新，又支持了四种数据类型：

### <font style="background-color:#FBDE28;">ZSet用过吗（介绍一下Zset）（字节）</font>
ZSet 就是一个自带排序功能的 Set。

它里面存的元素是不重复的，但是每个元素都会额外绑一个叫 Score 的分数，Redis 就是靠这个分数来给元素们从小到大排队的。

说到**底层实现**，ZSet 其实是很聪明的，它有两套结构切着用：  

当数据量不大、字符串也不长的时候，它用的是压缩列表（在 Redis 7.0 之后换成了更紧凑的 Listpack）。这时候数据都是紧紧挨着存的，主打就是一个省内存。

等数据量一大，它就会切成**‘哈希表 + 跳表’**的组合拳。

+ **哈希表**用来存元素和分数的映射关系，这样比如我要单独查某篇文章的得分，O(1) 的时间就能拿到。
+ **跳表**呢，主要就是用来维护这个排行榜的顺序。相比于普通的链表，它加了多层索引，这样不管我是插入、删除，还是做范围查找，时间复杂度基本都能稳在 O(logN)。

这里也多说一句，之前也有人问过我**为什么 Redis 用跳表不用红黑树**。

我个人的理解是：

一方面跳表做范围查找特别爽，找到起点后，直接顺着最底层的链表往后‘捞’就行了，如果是红黑树还得搞中序遍历，挺折腾的；

另一方面就是跳表的实现其实比平衡树简单很多，内存占用也比较灵活。

**结合到我自己的项目里**，在做那个汇率资讯系统的时候 ，我就直接用了 Redis ZSet 来实现点赞排行榜 。

具体做法就是把**文章 ID 当作 Member**，**点赞数当作 Score**。每次有用户点赞，我就起个 `ZINCRBY` 让分数加一；

前端要拉取首页 Top 10 热榜的时候，我就直接用 `ZREVRANGE` 取前十名。

这比直接去 MySQL 里 `ORDER BY` 加 `LIMIT` 性能高太多了，而且完美避开了高并发下的数据库压力。

### <font style="color:rgb(51, 51, 51);">提到了ZSet用于排行榜，如果相同分数但是我想让先达到这个分数的人排在前面，如何设计（字节）</font>
关于这种‘点赞数优先，时间先到者优先’的双维度排序，我的核心思路是**把两个维度的值，通过数学运算压缩到一个 Score 里**。  

因为我们在业务里通常是用 `ZREVRANGE` 取倒序排行的，也就是说 **Score 越大，排名越靠前**。

点赞数越大分越高，这个很直观；

但时间维度是相反的，**时间越早，时间戳越小，但我们希望它排名更靠前（分越高）**。

所以，具体的设计方案是这样的：

**1. 设计复合分数 (Composite Score)** 我们可以把 Score 拆成**整数和小数**两部分：

+ **整数部分：** 代表点赞数（或者基础积分）。
+ **小数部分：** 代表**时间戳的‘反转值’**。

在业务需求里，我们的双维度排序规则是：

+ **第一维度（点赞数）：** 越多越好（值越大，排越前）。
+ **第二维度（达成时间）：** 越早越好（**时间戳越小**，排越前）。

关于这个小数部分的设计，一开始是想的取倒数，但是后面在网上调研了一下，发现取倒数会导致精度丢失 （太小了） 。

所以定义一个未来的极限时间戳（比如 2099 年的某一天，或者直接用 `9999999999`）。 

然后用 `(极限时间戳 - 当前获取该分数的绝对时间戳) / 10^10`

计算公式是： `小数部分 = (极限时间戳 - 当前获取该分数的绝对时间戳) / 10^10`

这样一来，如果两个人的点赞数都是 100，谁先达到这个分数（当前时间戳越小），`(极限时间戳 - 当前时间戳)` 的值就越大，他的最终 Score 就会是 `100.089...`，而后达到的人可能是 `100.088...`。

在 `ZREVRANGE` 倒序时，先达到的人自然就排在前面了。

**每次执行 **`**ZADD**`** 或 **`**ZINCRBY**`**，元素都会在跳表中自动定位到正确的位置，数据时刻处于有序状态**。

+ 对于更新（玩家分数变动）：

    - $ \text{ZSet} $ 最大的优势在于，**数据在写入时就已经排好序了**。我们查询时不需要做任何计算，只是去内存里把**已经排好**的列表**取出来**，所以能做到绝对的“实时”。

$ \text{Redis} $ 负责实时读写，但数据必须持久化。

在代码层面，我们会重点关注我们会重点关注：

+ **连接池**：使用高效的 Redis 客户端（如 go-redis），并合理配置连接池参数，避免连接成为瓶颈。
+ **Pipeline / 事务**：在某些需要先读取、再计算的场景（虽然不是简单递增），可以使用 Pipeline 减少 RTT，或使用 Lua 脚本保证原子性。

### Redis 中 set和zset区别是什么？
核心区别：

+ 有序性：$ \text{Set} $ 完全无序，底层就是哈希表。而 $ \text{ZSet} $ 最大的不同是引入了 $ \text{Score} $，它在底层是通过 跳表 ($ \text{Skip List} $)  来实现的，**跳表保证了数据在内存中时刻都是有序的**
+ 操作特性：**ZSet多了基于分数的排序、范围查询、排名计算等命令，Set只有集合本身的交并差等操作**
+ 适用场所：**Set用于去重且无需排序的场景、ZSet用于排序、排名或权重相关的场景**

### Zset 底层是怎么实现的？
Zset 类型的底层数据结构是由**压缩列表或跳表**实现的：

+ 如果有序集合的**元素个数小于 128 个**，并且**每个元素的值小于 64 字节时**，Redis 会使用**压缩列表**作为 Zset 类型的底层数据结构；
+ 如果有序集合的元素不满足上面的条件，Redis 会使用**跳表和哈希表**作为 Zset 类型的底层数据结构；

**跳表：核心存储结构，负责维护成员的有序性，支持高效的范围查询（如 **`ZRANGE`**）、插入、删除操作（时间复杂度均为 O (logN)）**

**哈希表：辅助结构，键是 Zset 的成员（member），值是对应的跳表节点指针，负责快速映射 “成员→分数（score）+ 跳表节点”，支持 O (1) 时间复杂度的成员查询（如 **`ZSCORE`**）、存在性判断（如 **`ZEXISTS`**）。**

### 跳表是怎么实现的？
链表在查找元素的时候，要逐一查找，效率低，于是出现了跳表。

**跳表是在链表基础上改进过来的，实现了一种「多层」的有序链表**，这样的好处是能快读定位数据。

层级为3的跳表：

![](https://cdn.nlark.com/yuque/0/2026/png/50467087/1772306829516-c2c03ec8-60eb-4040-a264-2d3c41c65fd6.png)

头节点L0/L1/L2,分别指向不同层级的节点，然后每个层级的节点通过指针连接起来：

+ L0 层级共有 5 个节点，分别是节点1、2、3、4、5；
+ L1 层级共有 3 个节点，分别是节点 2、3、5；
+ L2 层级只有 1 个节点，也就是节点 3 。

当数据量很大时，跳表的查找复杂度就是 O(logN)。

### 跳表是怎么设置层高的？
跳表在创建节点时，生成范围[0,1]的**随机数**,如果这个随机数**小于 0.25**（相当于概率 25%），那么**层数就增加 1 层**,然后**继续生成下一个随机数**，直到随机数的结果**大于 0.25 结束**，最终确定该节点的层数。

### Redis为什么使用跳表而不是用B+树?
Redis 是内存数据库，**跳表在实现简单性、写入性能、内存访问模式等方面的综合优势**

| 维度 | 跳表优势 | B+**劣势** |
| --- | --- | --- |
| **内存访问** | 符合CPU缓存局部性,指针跳转更加高效 | 节点机构复杂,缓存不友好 |
| **实现复杂度** | 代码简洁，无复杂平衡操作 | 节点分裂/合并逻辑复杂，代码量大 |
| **写入性能** | 插入/删除仅需调整局部指针 | 插入可能触发递归节点分裂，成本高 |
| **内存占用** | 结构紧凑，无内部碎片 | 节点预分配可能浪费内存 |


**B+ 树**：**磁盘友好**，**节点填充率高**，**范围查询高效**

**跳表**：**内存友好**，**简单灵活**，**概率平衡**

**Redis 是内存数据库**，数据完全存储在内存中，不需要优化磁盘I/O，因此 B+ 树的磁盘友好特性对 Redis 意义不大。而跳表的内存访问模式更优，更适合高频的内存操作。

### 压缩列表是怎么实现的？
压缩列表是 **Redis 为了节约内存而开发的**，它是**由连续内存块组成的顺序型数据结构**，有点类似于数组。

压缩列表是一种**顺序存储的线性结构**，本质是一块连续的内存字节数组，没有链表的前后指针开销，所以内存占用极低。

但代价是插入 / 删除需要移动字节，时间复杂度 O (N)，因此仅适合 “元素少、数据短” 的场景（比如 Zset 元素≤128 个、member 长度≤64 字节）。

![](https://cdn.nlark.com/yuque/0/2026/png/50467087/1772306829468-60f341e5-1f77-4555-9404-f6c475f19a89.png)

压缩列表的结构分为 3 部分：

**表头（zlbytes）→ 数据节点（entry）→ 表尾标记（zlend）**

压缩列表在表头有三个字段:

+ zlbytes,记录整个压缩列表占用对**内存字节数**；
+ zltail,记录压缩列表「尾部」节点距离起始地址由多少字节，也就是**列表尾的偏移量**；
+ zllen,记录压缩列表包含的**节点数量**；

在压缩列表中，如果我们要查找定位第一个元素和最后一个元素，可以通过表头三个字段（zllen）的长度直接定位，复杂度是 O(1)。

而**查找其他元素时，就没有这么高效了，只能逐个查找，此时的复杂度就是 O(N) 了，因此压缩列表不适合保存过多的元素。**

### 介绍一下 Redis 中的 listpack(紧凑列表)
listpack（紧凑列表）是 Redis 7.0 版本正式引入的**高效紧凑存储结构**，核心定位是**替代老旧的压缩列表（ziplist）**，解决 ziplist 的 “连锁更新” 痛点，同时保留 “内存紧凑” 的核心优势，目前已逐步成为 Hash、Zset 等结构的小体量存储方案（未来会完全替代 ziplist）。

ziplist 虽然省内存，但存在一个致命缺陷 ——**连锁更新**（当一个节点长度变化时，后续所有节点的 `prevlen` 字段都可能需要重新编码，导致 O (N²) 时间复杂度）。

listpack 是 Redis 对 “紧凑存储” 的优化升级，核心是「保留 ziplist 的省内存优势，解决其连锁更新的致命缺陷」。它的结构更简洁、编码更高效、操作更稳定，目前已成为 Hash、Zset 小体量存储的首选，未来会完全替代 ziplist，成为 Redis 紧凑存储的核心结构。

### 哈希表是怎么扩容的？
扩容核心是 **“渐进式扩容”**—— 避免一次性迁移大量数据导致主线程阻塞，通过分批次迁移数据，平衡性能和可用性。

进行 rehash 的时候，需要用上 2 个哈希表了

<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2026/png/50467087/1772306829467-da2ba46e-7178-49a1-9b69-c94e73f3a13e.png)

在正常服务请求阶段，插入的数据，都会写入到「哈希表 1」，此时的「哈希表 2 」 并没有被分配空间。

随着数据逐步增多，触发了 rehash 操作，这个过程分为三步：

+ 给「哈希表 2」 分配空间，一般会比「哈希表 1」 大 2 倍；
+ 将「哈希表 1 」的数据迁移到「哈希表 2」 中；
+ 迁移完成后，「哈希表 1 」的空间会被释放，并把「哈希表 2」 设置为「哈希表 1」，然后在「哈希表2」 新创建一个空白的哈希表，为下次 rehash 做准备。

渐进式 rehash 步骤如下：

+ 给「哈希表 2」 分配空间；
+ **在 rehash 进行期间，每次哈希表元素进行新增、删除、查找或者更新操作时，Redis 除了会执行对应的操作之外，还会顺序将「哈希表 1 」中索引位置上的所有 key-value 迁移到「哈希表 2」 上**；
+ 随着处理客户端发起的哈希表操作请求数量越多，最终在某个时间点会把「哈希表 1 」的所有 key-value 迁移到「哈希表 2」，从而完成 rehash 操作。

### 哈希表扩容的时候，有读请求怎么查？、
查找一个 key 的值的话，先会在「哈希表 1」 里面进行查找，如果没找到，就会继续到哈希表 2 里面进行找到。

### **String 是使用什么存储的?为什么不用 c 语言中的字符串?**
Redis 的 String 字符串是用 **SDS 数据结构**存储的。

<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2026/png/50467087/1772306829483-d5afdda3-d64f-4a2b-81b9-d7eac94c592e.png)

Redis 的 SDS 结构在原本字符数组之上，增加了三个元数据：len、alloc、flags

```plain
struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;     // 已使用的字节数（字符串实际长度）
    uint8_t alloc;   // 总分配的字节数（len + 未使用的 free）
    unsigned char flags;  // 标记 SDS 类型（区分不同长度的 len/alloc，如 8 位/16 位/32 位/64 位）
    char buf[];      // 字符数组（存储实际数据，最后仍以 \0 结尾，兼容 C 函数）
};
```

SDS 通过以下设计，完美解决 C 字符串的缺陷：

**O (1) 时间获取长度**：`len` 字段直接记录字符串实际长度，获取长度只需读取 `len`，无需遍历，满足 Redis 高频查询长度的需求（如 `STRLEN` 命令）。

**杜绝缓冲区溢出**：修改字符串时（如 `sdscat` 拼接），SDS 会先检查 `alloc - len`（剩余可用空间）是否足够

**优化内存重分配：减少次数 + 节省空间**

#### C 语言字符串的缺陷（为什么不用）
获取长度效率极低（O (N) 时间）

存在缓冲区溢出风险

内存重分配频繁，性能损耗大

不支持二进制数据

### Redis为什么那么快
官方使用基准测试的结果是，**单线程的 Redis 吞吐量可以达到 10W/每秒**

之所以 Redis 采用单线程（网络 I/O 和执行命令）那么快，有如下几个原因：

+ Redis的**大部分操作都是在内存中完成的,采用了更加高效的数据结构**
+ Redis**采用单线程模型**可以**避免了多线程之间的竞争**,省去了多线程切换带来的时间和性能上的开销
+ Redis 采用了 **I/O 多路复用机制 **处理大量的客户端 Socket 请求,IO 多路复用机制是指一个线程处理多个 IO 流，就是我们经常听到的 select/epoll 机制。

### <font style="color:rgb(51, 51, 51);">redis在任何时候都是单线程的吗？</font>
**不是的，这是一个非常经典的误区。**

其实我们常说的‘Redis 是单线程’，**仅仅是指它执行用户命令（比如读写 KV 数据）的那个核心事件循环是单线程的**。

但在整个 Redis 的生命周期和架构里，它早就不是纯单线程了。

具体来说，可以分为三个层面：

**第一层：核心命令执行始终是单线程。**

 因为 Redis 是基于内存操作的，瓶颈在网络和内存，不在 CPU。单线程执行反而完美避开了多线程的上下文切换和锁竞争开销，保证了像 `INCR` 或者执行 Lua 脚本时的绝对原子性。

**第二层：脏活累活早就交给了后台多线程（Redis 4.0+）。**

为了防止单线程被卡死，Redis 引入了 **BIO（Background I/O）后台线程**。

比如我想删掉一个几百万元素的集合，用 `DEL` 会阻塞主线程，但我用 `UNLINK` 命令，主线程就只负责解除引用，真正释放内存的重活儿是由后台线程异步去干的。

另外，AOF 的异步刷盘（fsync）也是由专门的后台多线程去完成的。

**第三层：网络请求解析全面拥抱多线程（Redis 6.0+）。**

随着网络硬件越来越快，Redis 发现主线程的时间都浪费在从 Socket 读数据和解析协议上了。

所以 Redis 6.0 引入了**网络 I/O 多线程**。

它会派多个线程去并发地**读取和解析客户端的网络请求**，**但解析完之后，依然是排好队交给那个唯一的主线程去执行**，执行完再交给多线程并发写回响应。

**总结一句话就是：**

**Redis 是‘单线程计算，多线程网络 I/O，外加多线程后台清理’的复合架构。**

### Redis哪些地方使用了多线程
我们常说的“Redis 是单线程”，**仅仅是指 Redis 处理网络请求和执行 KV 键值对读写命令的核心主事件循环（Event Loop）是单线程的**。  

**因为 Redis 的瓶颈从来不在 CPU，而在内存访问速度和网络 I/O。**

**单线程执行命令，完美避开了多线程上下文切换的开销，以及复杂的锁竞争（比如在做 **`**ZINCRBY**`** 或者执行 Lua 脚本时，天生保证了操作的原子性）。**

Redis 的核心处理逻辑（命令执行）是单线程的，但在**特定辅助功能**中引入了多线程，主要用于解决 “主线程阻塞” 问题，提升整体性能。

这些多线程场景集中在 **“IO 密集型” 或 “耗时操作”** 上，避免占用主线程资源。

多线程主要用于：

+ **后台 I/O 线程**：处理异步文件关闭、AOF 刷盘、内存释放等耗时操作；
+ **网络 IO 多线程**（6.0+）：并行处理网络读写，提升高并发下的吞吐量；
+ **集群后台线程**：处理集群槽位迁移、状态同步等维护操作。

### Redis怎么实现的io多路复用？
为什么 Redis 中要使用 I/O 多路复用这种技术呢？

因为 Redis 是跑在「单线程」中的，**所有的操作都是按照顺序线性执行的，但是由于读写操作等待用户输入 或 输出都是阻塞的，所以 I/O 操作在一般情况下往往不能直接返回，这会导致某一文件的 I/O 阻塞导，致整个进程无法对其它客户提供服务**。

而 I/O 多路复用就是为了解决这个问题而出现的。

为了**让单线程(进程)的服务端应用同时处理多个客户端的事件**，Redis 采用了 IO 多路复用机制。

这里“多路”指的是**多个网络连接客户**端，“复用”指的是**复用同一个线程(单进程)。**

I/O 多路复用其实是**使用一个线程来检查多个 Socket 的就绪状态**，**在单个线程中通过记录跟踪每一个 socket（I/O流）的状态来管理处理多个 I/O 流。**

如下图是 Redis 的 I/O 多路复用模型：

![](https://cdn.nlark.com/yuque/0/2026/png/50467087/1772306829480-2fa7910e-b4a5-4882-af71-75dfe96c89a5.png)

+ **一个 socket 客户端与服务端连接时，会生成对应一个套接字描述符**(套接字描述符是文件描述符的一种)，**每一个 socket 网络连接其实都对应一个文件描述符。**
+ **多个客户端与服务端连接时，Redis 使用 I/O 多路复用程序 将客户端 socket 对应的 FD 注册到监听列表(一个队列)中。**当客服端执行 read、write 等操作命令时，I/O 多路复用程序会将命令封装成一个事件，并绑定到对应的 FD 上。
+ **文件事件处理器使用 I/O 多路复用模块同时监控多个文件描述符（fd）的读写情况，当 accept、read、write 和 close 文件事件产生时，文件事件处理器就会回调 FD 绑定的事件处理器进行处理相关命令操作。**

### Redis的网络模型是怎样的？
**Redis 的网络模型**主要基于 **单线程 Reactor 模型**（也可以理解为 **多路复用 + 事件驱动机制**）

**单 Reactor 单进程的方案因为全部工作都在同一个进程内完成**，所以实现起来比较简单，不需要考虑进程间通信，也不用担心多进程竞争。

但是，这种方案存在 2 个缺点：

+ 第一个缺点，**因为只有一个进程，无法充分利用 多核 CPU 的性能**；
+ 第二个缺点，**Handler 对象在业务处理时，整个进程是无法处理其他连接的事件的，如果业务处理耗时比较长，那么就造成响应的延迟**；所以，**单 Reactor 单进程的方案不适用计算机密集型的场景，只适用于业务处理非常快速的场景。**

Redis 是由 C 语言实现的，在 Redis 6.0 版本之前采用的正是「单 Reactor 单进程」的方案，因为 Redis 业务处理主要是在内存中完成，操作的速度是很快的，性能瓶颈不在 CPU 上，所以 Redis 对于命令的处理是单进程的方案。

到 Redis 6.0 之后，就将网络IO的处理改成多线程的方式了，目的是为了这是因为随着网络硬件的性能提升，Redis 的性能瓶颈有时会出现在网络 I/O 的处理上。

所以为了提高网络 I/O 的并行度，Redis 6.0 对于网络 I/O 采用多线程来处理。但是对于命令的执行，Redis 仍然使用单线程来处理，所以大家不要误解 Redis 有多线程同时执行命令。

## 三.事务
### 如何实现redis 原子性？
**Redis 执行一条命令时本身就是具备原子性的。**

因为** Redis 的命令是由单线程来处理的**，不存在多线程竞争问题，也就不会出现部分执行或中途被打断的情况。

但是**如果要保证多条命令的原子性，比如两条或多条命令要么都执行，要么都不执行**，那就需要借助 **Lua 脚本**。

我们可以**把多条操作写进一个 Lua 脚本中，Redis 会把整个脚本当成一个整体去执行，在执行过程中不会被其他命令插入或打断，因此能保证脚本中所有操作的原子性**。

举个例子，

比如我们在实现 **分布式锁解锁** 的时候，**需要先判断当前锁是不是自己加的，如果是才删除锁，这其实是两个操作。**  
 为了避免在判断和删除之间被其他请求插入，我们会通过 Lua 脚本来实现，比如

```plain
if redis.call('get', KEYS[1]) == ARGV[1] then
  return redis.call('del', KEYS[1])
else
  return 0
end
```

这样就能保证判断和删除这两个操作的原子性。

Redis 也有自己的事务机制，通过 `MULTI` 开启，`EXEC` 执行，配合 `WATCH` 可以实现类似**乐观锁**的效果（监控某个 Key，如果被别人改了，事务就不执行）。   但是，Redis 的事务和 MySQL 的事务（ACID）有本质区别。

Redis 事务**不支持回滚（Rollback）**。如果在 `EXEC` 执行期间，某条命令因为数据类型不匹配报错了，Redis 会继续执行后面的命令，不会撤销前面已经成功的命令。所以为了稳妥，我个人在复杂的原子操作选型上，**更倾向于优先使用 Lua 脚本**。  

### 除了lua有没有什么也能保证redis的原子性？
除了 Lua 脚本以外，Redis 其实还有几种方式能在不同层面上保证 **原子性**，不过要分清楚它们的适用范围。

Redis事务（MULTI / EXEC）：

Redis 提供了事务机制，也可以在一定程度上保证多命令的原子性。

```plain
MULTI
DECR accountA
INCR accountB
EXEC
```

**特点：**

+ `MULTI` 开始事务，后续命令进入队列。
+ `EXEC` 执行时会一次性按顺序执行所有命令，中间不会被其他命令打断。
+ ⚠️ 但是需要注意：

  ✅ **总结：MULTI/EXEC 能保证执行时不被打断，但不能回滚**

    - Redis 事务 **不支持回滚（rollback）**。  
   如果其中一条命令执行失败，前面的命令仍然会生效。
    - 因此 Redis 的事务保证的是“**执行期间的原子性**”，不是“逻辑层面的原子一致性”。

WATCH乐观锁：

`WATCH` 是 Redis 提供的一种 **乐观锁机制**，能在一定程度上实现条件式原子更新。

```go
WATCH key
val = GET key
if val >= 10:
    MULTI
    DECRBY key 10
    EXEC
```

**机制说明：**

+ `WATCH` 会监控一个或多个 key；
+ 如果在执行 `EXEC` 前这些 key 被别的客户端修改了，事务就会失败；
+ 这样可以防止并发条件下的数据竞争。

✅ **总结：WATCH + MULTI/EXEC 组合，适合需要“先读后改”的场景，实现逻辑层面的原子更新。**

除了 Lua 脚本外，Redis 还可以通过单线程模型保证单命令的原子性，通过 `MULTI/EXEC` 事务机制和 `WATCH` 乐观锁实现多命令的原子性控制。而在分布式场景下，可以用 RedLock 算法保证多节点间的原子性。

## 四.日志
### Redis有哪2种持久化方式？分别的优缺点是什么？
Redis 的读写操作都是在内存中，所以 Redis 性能才会高，**但是当 Redis 重启后，内存中的数据就会丢失，那为了保证内存中的数据不会丢失，Redis 实现了数据持久化的机制**，**这个机制会把数据存储到磁盘，这样在 Redis 重启就能够从磁盘中恢复原有的数据。**

Redis 共有两种数据持久化的方式：

+ **AOF 日志：每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里；**
+ **RDB 快照：将某一时刻的内存数据，以二进制的方式写入磁盘；**

**AOF 持久化（追加日志方式）**

**AOF 它会把 Redis 执行的每一条写命令都以文本的形式追加记录到日志文件里。**

其实 AOF 的本质就是 Write-Ahead Log (WAL)，如果你研究过分布式系统底层的日志复制原理，对这种机制肯定非常亲切。

**原理：**  
 AOF 会把 **每一次写命令**（如 `SET`、`INCR`、`DEL`）都以文本日志的方式追加到文件（`appendonly.aof`）末尾。  
 Redis 重启时会重新执行这些命令来恢复数据。

这里以「_set name xiaolin_」命令作为例子，Redis 执行了这条命令后，记录在 AOF 日志里的内容如下图：

![](https://cdn.nlark.com/yuque/0/2026/webp/50467087/1772306829786-6f3f1aa8-0579-4051-a96c-080adeba77ea.webp)

**Redis 提供了 3 种写回硬盘的策略， 在 Redis.conf 配置文件中的 appendfsync 配置项可以有以下 3 种参数可填：**

+ `always`：每次写操作都立即同步到磁盘（最安全，性能最低）。
+ `everysec`：每秒同步一次（默认，性能与安全性平衡）。
+ `no`：由操作系统决定何时写入磁盘（性能高，但可能丢失更多数据）。

**RDB 持久化（快照方式）**：

 RDB 就像是给当前内存里的数据拍了一张‘全景照片’，把某一时刻的全量数据生成一个经过压缩的二进制文件（`.rdb`）。  

+ **优点（主打恢复快）：** 因为它是紧凑的二进制文件，非常适合做灾备和全量复制。当 Redis 重启时，直接把整个文件加载进内存就行，**数据恢复的速度极快**，远超 AOF。
+ **缺点（丢数据风险与 Fork 阻塞）：** 首先是会**丢失两次快照之间的数据**（比如每 5 分钟拍一次，宕机最多丢 5 分钟数据）。其次，RDB 在执行 `bgsave` 时，主线程会 `fork()` 出一个子进程去写磁盘。虽然有 OS 层面的 Copy-On-Write（写时复制）技术兜底，但如果 Redis 实例内存极大（比如几十 GB），`fork()` 这个系统调用本身在复制页表时依然会造成主线程的瞬间阻塞。

**原理：**  
 RDB 是通过**保存某一时刻的内存数据快照**来实现的。  
 Redis 会定期把内存中的数据写入一个 `.rdb` 文件中（默认 `dump.rdb`），实现定时持久化。

**触发方式：**

+ 自动触发（在配置文件中通过 `save` 参数指定，例如：`save 900 1` 表示 900 秒内至少有 1 次修改就触发保存）；
+ 手动执行 `SAVE` 或 `BGSAVE` 命令。

**总结：**

**Redis 有两种持久化方式：RDB 和 AOF。**  
**RDB 是定期生成数据快照，优点是性能好、恢复快，但可能丢数据；**  
**AOF 是将写命令追加到日志中，数据更安全，但文件更大、恢复慢。**  
**一般线上会 同时开启 RDB 和 AOF，用 RDB 备份、AOF 保证实时性。**

## 五.缓存淘汰和过期删除
### 过期删除策略和内存淘汰策略有什么区别？
区别：

+ **内存淘汰策略**是在**内存满了**的时候，redis 会触发内存淘汰策略，来淘汰一些不必要的内存资源，以腾出空间，来保存新的内容
+ **过期键删除策略**是**将已过期的键值对进行删除**，Redis 采用的删除策略是**惰性删除+定期删除**。

### 介绍一下Redis 内存淘汰策略
Redis 是一个基于内存的数据库，内存容量是有限的。

当内存达到上限时（由 `maxmemory` 参数控制），Redis 就需要**淘汰一部分数据**来给新的数据腾出空间，这个过程就叫 **内存淘汰（Eviction）策略**。

只有在以下两种情况下，Redis 才会触发内存淘汰机制：

1. 设置了 `maxmemory` 参数；
2. 写入新数据时内存超过上限。

**Redis 内存淘汰策略共有八种，这八种策略大体分为「不进行数据淘汰」和「进行数据淘汰」两类策略。**

**默认的策略是 **`**noeviction**`**（不淘汰）。这种模式下，Redis 拒绝接受任何新的写请求并报错，只保证读操作正常。这一般是在把 Redis 当作纯数据库（绝不能丢数据）时才用，但在我们做微服务缓存时，肯定要改成其他的动态淘汰策略。**

为了方便记忆和选型，我把剩下的 7 种策略看作是一个组合矩阵。 

**首先是划分‘淘汰范围’（2个维度）：**

+ `**volatile-***`**：只从设置了过期时间（TTL）的 Key 里面挑。**
+ `**allkeys-***`**：从所有的 Key 里面挑（不管有没有设过期时间）。**

**然后是划分‘淘汰算法’（3个维度）：**

+ `***-lru**`** (Least Recently Used)****：挑出****最久没被访问过****的 Key 淘汰。这是最经典的缓存策略。**
+ `***-lfu**`** (Least Frequently Used)****：挑出****访问频率最低****的 Key 淘汰。这是 Redis 4.0 之后引入的。**
+ `***-random**`**：纯随机淘汰（一般不用）。**

**另外再加上 1 个特例：**

+ `**volatile-ttl**`**：专门挑那些最快要过期的 Key 提前淘汰掉。**

****

<!-- 这是一张图片，ocr 内容为： -->
![](https://cdn.nlark.com/yuque/0/2026/png/50467087/1772306829914-022d718e-0a1a-4eab-b07a-d5397048799c.png)

不进行数据淘汰：

+ **noeviction**（默认策略）  
   不会删除任何数据。  
   当内存满时，如果有写操作，会直接报错（读操作仍可执行）。

📌 **适用场景**：希望 Redis 保证数据完整性，如缓存和数据库共用场景。

进行数据淘汰：

+ 只淘汰“设置了过期时间”的键：
1. **volatile-lru**：  
  从设置了过期时间的 key 中，**优先淘汰最近最少使用（LRU）** 的数据。
2. **volatile-lfu**（Redis 4.0+）：  
  从设置了过期时间的 key 中，**优先淘汰使用频率最低（LFU）** 的数据。
3. **volatile-ttl**：  
  从设置了过期时间的 key 中，**优先淘汰快要过期的** 数据。
4. **volatile-random**：  
  从设置了过期时间的 key 中，随机删除一些 key。

📌 **适用场景**：主要用于做缓存，有部分 key 有过期时间。

+ 在所有键中进行淘汰
1. **allkeys-lru**：  
  从所有 key 中淘汰最近最少使用的 key。  
  👉 这是缓存系统中最常用的策略。
2. **allkeys-lfu**：  
  从所有 key 中淘汰使用频率最低的 key。  
  👉 相比 LRU，更能反映真实访问热点。
3. **allkeys-random**：  
  从所有 key 中随机淘汰一些 key。

📌 **适用场景**：所有 key 都可作为缓存，没有明确的过期时间。

| 策略 | 范围 | 淘汰依据 | 说明 |
| --- | --- | --- | --- |
| noeviction | 无 | 不淘汰 | 默认策略，内存满时报错 |
| volatile-lru | 仅过期键 | 最近最少使用 | 优先删除冷数据 |
| allkeys-lru | 所有键 | 最近最少使用 | 缓存常用策略 |
| volatile-lfu | 仅过期键 | 使用频率最低 | 适合访问规律型数据 |
| allkeys-lfu | 所有键 | 使用频率最低 | Redis 4.0 以后推荐 |
| volatile-ttl | 仅过期键 | 即将过期 | 优先删除快到期的键 |
| volatile-random | 仅过期键 | 随机 | 随机删除部分键 |
| allkeys-random | 所有键 | 随机 | 简单快速但效果一般 |


如果 Redis 主要**用于做缓存**（如热点数据缓存）：  
 ✅ 推荐使用 `allkeys-lru` 或 `allkeys-lfu`。

如果**只是部分数据设置了过期时间：**  
 ✅ 推荐使用 `volatile-lru`。

如果**希望数据不能被自动删除：**  
 ✅ 保持默认 `noeviction`。

简洁版：

Redis 有两种主要的内存淘汰机制配置方式。  
 当内存达到上限（`maxmemory`）时，Redis 会根据配置的策略删除部分数据。

常见的策略有 8 种，默认是 `noeviction`，也就是内存满了就直接报错。  
 最常用的两种是：

+ **allkeys-lru**：淘汰所有 key 中最近最少使用的；
+ **allkeys-lfu**：淘汰使用频率最低的。

另外也有只淘汰设置过期时间键的策略，比如 `volatile-lru`、`volatile-ttl` 等。  
 实际项目里，一般会用 `allkeys-lru` 做缓存淘汰，因为它能自动清理冷数据，效果最好

### Redis 的 LRU 是怎么实现的？
Redis 并没有严格实现真实的 LRU 算法，而是做了一个 **近似 LRU（Approximate LRU）** 的优化版本。

具体来说，当启用 `allkeys-lru` 或 `volatile-lru` 策略时，Redis 会在内存不足时：

1. **随机采样若干个键（默认 5 个，可通过 **`maxmemory-samples`** 调整）；**
2. 在这几个键中找到**最近最少使用**的那个；
3. 淘汰它。

Redis 的 LRU 是一种“采样近似算法”，通过随机选取部分 key 并淘汰其中访问最少的来实现性能与效果的平衡。  
如果想更精准地反映访问热度，可以使用 Redis 4.0 以后的 LFU 策略

### 介绍一下Redis过期删除策略
Redis 选择「惰性删除+定期删除」这两种策略配和使用，以求在合理使用 CPU 时间和避免内存浪费之间取得平衡。

也就是说：	

1. **当访问一个 key 时，会做惰性检查**；
2. **同时 Redis 在后台定期执行过期扫描，主动清理一部分过期 key；**
3. 如果仍然存在未清理的过期数据，当内存不足时，还会被 **内存淘汰策略** 清除。

这样既保证了性能，又避免了内存泄漏。

**惰性删除（被动删除）：**

当客户端访问某个 key 时，Redis 才会检查它是否过期：

+ 如果已过期，就在访问时删除并返回空；
+ 如果未过期，则正常返回。

**适用于访问热点数据较多的场景**

Redis 的**惰性删除策略**由 db.c 文件中的 expireIfNeeded 函数实现，代码如下：

```plain
int expireIfNeeded(redisDb *db, robj *key) {
    // 判断 key 是否过期
    if (!keyIsExpired(db,key)) return 0;
    ....
    /* 删除过期键 */
    ....
    // 如果 server.lazyfree_lazy_expire 为 1 表示异步删除，反之同步删除；
    return server.lazyfree_lazy_expire ? dbAsyncDelete(db,key) :
                                         dbSyncDelete(db,key);
}
```

Redis 在访问或者修改 key 之前，都会调用 **expireIfNeeded 函数对其进行检查，检查 key 是否过期**：

+ 如果过期，则删除该 key，至于选择异步删除，还是选择同步删除，根据 lazyfree_lazy_expire 参数配置决定（Redis 4.0版本开始提供参数），然后返回 null 客户端；
+ 如果没有过期，不做任何处理，然后返回正常的键值对给客户端；

**定期删除（主动批量删除）**：

**Redis 会定期（默认每 100ms）随机抽取一些设置了过期时间的 key，检查它们是否过期，如果过期则删除。**

### Redis的缓存失效会不会立即删除？
在过期 key 比较多的情况下，删除过期 key 可能会占用相当一部分 CPU 时间，在内存不紧张但 CPU 时间紧张的情况下，将 CPU 时间用于删除和当前任务无关的过期键上，无疑会对服务器的响应时间和吞吐量造成影响。

所以，定时删除策略对 CPU 不友好。

总之：

**Redis 的缓存失效不会立即删除，而是延迟删除，通过惰性删除和定期删除两种机制来实现平衡性能与内存的效果。**

## 六.集群
### Redis主从同步中的增量和完全同步怎么实现？
在 Redis 主从复制（Replication）中，<font style="background-color:#FBDE28;">从节点要和主节点保持数据一致</font>，复制过程分为两种情况：

+ **全量（完全）同步**：**从节点第一次连接主节点或数据差异太大时使用**。
+ **增量同步**：**主从短暂断开后重新连接时使用，只同步丢失的部分数据。**

**完全同步（Full Resynchronization）**

完全同步发生在以下几种情况：

+ **初次同步**：当一个从服务器（slave）首次连接到主服务器（master）时，会进行一次完全同步。
+ **从服务器数据丢失**：如果从服务器数据由于某种原因（如断电）丢失，它会请求进行完全同步。
+ **主服务器数据发生变化**：如果从服务器长时间未与主服务器同步，导致数据差异太大，也可能触发完全同步。

主从服务器间的第一次同步的过程可分为三个阶段：

第一阶段是建立链接、协商同步；

第二阶段是主服务器同步数据给从服务器；

第三阶段是主服务器发送新写操作命令给从服务器。

**实现过程**：

1. **从服务器发送SYNC命令**：从服务器向主服务器发送`SYNC`命令，请求开始同步。
2. 主服务器生成RDB快照：接收到SYNC命令后，主服务器会保存当前数据集的状态到一个临时文件，这个过程称为RDB（Redis Database）快照。
3. **传输RDB文件**：主服务器将生成的RDB文件发送给从服务器。
4. **从服务器接收并应用RDB文件**：从服务器接收RDB文件后，会清空当前的数据集，并载入RDB文件中的数据。
5. **主服务器记录写命令**：在RDB文件生成和传输期间，主服务器会记录所有接收到的写命令到replication backlog buffer。
6. **传输写命令**：一旦RDB文件传输完成，主服务器会将replication backlog buffer中的命令发送给从服务器，从服务器会执行这些命令，以保证数据的一致性。

**增量同步（Partial Resynchronization）**

**增量同步允许从服务器从断点处继续同步，而不是每次都进行完全同步。**

它基于PSYNC命令，使用了运行ID（run ID）和复制偏移量（offset）的概念。

**实现过程：**

1. 主从节点各自维护一个 **复制偏移量 offset**；
2. 主节点会把最近的写命令写入 **复制积压缓冲区**（默认 1MB，可配置）；
3. 当从节点断线重连后，会发送 `PSYNC <runid> <offset>` 给主节点；
4. 主节点检查该 offset 是否还在积压缓冲区中：
    - ✅ 在缓冲区中：直接把缺失的数据从缓冲区发给从节点（增量同步）。
    - ❌ 不在缓冲区中：执行一次完全同步。

### redis主从和集群可以保证数据一致性吗 ？
redis 主从和集群在CAP理论都属于AP模型，**即在面临网络分区时选择保证可用性和分区容忍性，而牺牲了强一致性。**

这意味着在网络分区的情况下，Redis主从复制和集群可以继续提供服务并保持可用，但可能会出现部分节点之间的数据不一致。

**那如果我想让 Redis 更一致，有什么办法？**

可以考虑使用 **WAIT 命令**（等待从节点确认写入）、或者用 **Redis Sentinel** + **主从延迟监控**，来在一定程度上增强一致性，但 Redis 本质上还是 AP 系统。

### CAP理论是什么？
**一致性，可用性，容错性**

三者无法同时满足，只能满足两个

经典取舍：

CP：一致性加容错性，<font style="color:rgba(0, 0, 0, 0.85);">牺牲可用性，保证分区下的数据一致。</font>

<font style="color:rgba(0, 0, 0, 0.85);">AP：可用性加容错性，牺牲强一致性，保证分区下服务可用（最终一致）。</font>

### 哨兵机制原理是什么？
Redis 的哨兵（Sentinel）机制主要用于实现<font style="background-color:#FBDE28;"> </font>**<font style="background-color:#FBDE28;">主从架构的自动故障转移和监控</font>**。  
它通过**<font style="background-color:#FBDE28;">持续监控主从节点的状态，一旦发现主节点宕机，会通过投票机制选举一个新的主节点，并通知其他从节点去复制新主节点</font>**，从而保证 Redis 的高可用性。

Redis Sentinel 是一套 **<font style="background-color:#FBDE28;">高可用监控与自动故障转移机制</font>**，它的核心功能有三点：

**监控（Monitoring）**

Sentinel 会定期**向主节点、从节点发送 **`**PING**`** 命令**，判断它们是否正常响应。

+ 如果某个节点在一定时间内没有响应，就被标记为 **主观下线（Subjectively Down, SDOWN）**。

**通知（Notification）**

当 Sentinel 发现主节点异常时，会通过 **发布订阅机制 通知其他 Sentinel 实例，共同确认主节点状态。**

**自动故障转移（Automatic Failover）**

+ 当多数 Sentinel 都认为主节点确实宕机（达成共识），该主节点被标记为 **客观下线（Objectively Down, ODOWN）**。
+ 哨兵集群会通过 **投票选举一个 Sentinel 领导者（Leader Sentinel）**，由它来执行故障转移：
    1. 从剩余的从节点中选出一个新的主节点；
    2. 让其他从节点改为复制新主节点；
    3. 通知客户端更新主节点信息。

整个过程是自动完成的，保证系统能在主节点宕机后继续提供服务。

### 哨兵机制的选主节点的算法介绍一下
当 Redis 主节点宕机后，Sentinel 会选举出一个哨兵领导者（Leader Sentinel）来执行故障转移。****  
领导者会从从节点中选择一个 **<font style="background-color:#FBDE28;">延迟最小、优先级最高、复制最完整</font>**<font style="background-color:#FBDE28;"> </font>的从节点作为新的主节点，然后让其他从节点去复制它。

### Redis集群的模式了解吗 优缺点了解吗
Redis 集群（Redis Cluster）是 Redis 官方提供的 **分布式部署方案**，<font style="background-color:#FBDE28;">通过 </font>**<font style="background-color:#FBDE28;">分片（sharding）</font>**<font style="background-color:#FBDE28;"> 将数据分布到多个节点上，并通过 </font>**<font style="background-color:#FBDE28;">主从复制 + 自动故障转移</font>**<font style="background-color:#FBDE28;"> 实现高可用</font>。

优点是能水平扩展、性能高、支持自动容错；  
缺点是事务支持不完善、数据迁移复杂、跨槽操作受限。

<font style="color:rgba(0, 0, 0, 0.85);">核心分为三类（主从、哨兵、Cluster）</font>

<font style="color:rgba(0, 0, 0, 0.85);">1.主从复制</font>

**<font style="color:rgb(0, 0, 0) !important;">架构</font>**<font style="color:rgba(0, 0, 0, 0.85);">：</font>

<font style="color:rgba(0, 0, 0, 0.85);">1 个主节点（Master）+ N 个从节点（Slave），主节点处理所有写请求，从节点通过 “全量复制（RDB）+ 增量复制（AOF / 复制缓冲区）” 同步主节点数据，仅处理读请求。</font>

**<font style="color:rgb(0, 0, 0) !important;">核心价值</font>**<font style="color:rgba(0, 0, 0, 0.85);">：</font>

<font style="color:rgba(0, 0, 0, 0.85);">读写分离、数据备份，解决单机读性能瓶颈。</font>

<font style="color:rgba(0, 0, 0, 0.85);">2.哨兵模式</font>

**<font style="color:rgb(0, 0, 0) !important;">架构</font>**<font style="color:rgba(0, 0, 0, 0.85);">：</font>

<font style="color:rgba(0, 0, 0, 0.85);">基于主从架构，引入 1 + 个哨兵节点（Sentinel），哨兵通过心跳检测监控主从节点状态，主节点故障时自动触发 “故障转移”（选举从节点升主、更新从节点配置、通知客户端）。</font>

**<font style="color:rgb(0, 0, 0) !important;">核心价值</font>**<font style="color:rgba(0, 0, 0, 0.85);">：</font>

<font style="color:rgba(0, 0, 0, 0.85);">解决主从模式 “人工故障转移” 的痛点，提升可用性（AP 方向）。</font>

<font style="color:rgba(0, 0, 0, 0.85);">3.</font><font style="color:rgb(0, 0, 0);">Redis Cluster（集群模式，官方推荐）</font>

## 七.场景
### 为什么使用redis？
使用 Redis 主要是为了 **提升系统性能和并发能力**。  
它把**热点数据放在内存中**，**读写速度非常快**（微秒级），可以极大减轻数据库压力。  
同时 Redis 还支持 **持久化、分布式锁、消息队列、计数器、缓存等多种场景**，是一个高性能、功能丰富的内存数据库。

| 场景 | Redis 功能 |
| --- | --- |
| 缓存热点数据 | 降低数据库压力 |
| 分布式锁 | 保证并发安全 |
| 限流计数 | 用 INCR / EXPIRE 实现接口限流 |
| 消息队列 | 用 List / Stream 实现异步解耦 |
| 排行榜系统 | 用 ZSet 维护排序数据 |
| Session 存储 | 用 Hash 存储用户状态 |


### 为什么redis比mysql要快？
+ **内存存储**：Redis 是基于内存存储的 NoSQL 数据库，而 MySQL 是基于磁盘存储的关系型数据库。由于内存存储速度快，Redis 能够更快地读取和写入数据，而无需像 MySQL 那样频繁进行磁盘 I/O 操作。
+ 简单数据结构：Redis 是基于键值对存储数据的，支持简单的数据结构（字符串、哈希、列表、集合、有序集合）。相比之下，MySQL 需要定义表结构、索引等复杂的关系型数据结构，因此在某些场景下 Redis 的数据操作更为简单高效。
+ 线程模型：Redis 采用单线程模型可以避免了多线程之间的竞争，省去了多线程切换带来的时间和性能上的开销，而且也不会导致死锁问题。

### 本地缓存和Redis缓存的区别?
**<font style="background-color:#FBDE28;">本地缓存</font>**<font style="background-color:#FBDE28;">（如 Guava Cache、Caffeine）存储在应用服务器内存中，访问速度最快，但 </font>**<font style="background-color:#FBDE28;">只对单机有效、无法共享</font>**<font style="background-color:#FBDE28;">。</font>

**Redis 缓存**是**独立的分布式缓存服务**，**<font style="background-color:#FBDE28;">多节点可共享数据，支持持久化和高可用</font>**<font style="background-color:#FBDE28;">，但网络开销更高、访问略慢。</font>

比如一个用户服务部署在三台机器上：

+ 使用本地缓存时，每台机器缓存自己的数据副本，更新时需要各自同步；
+ 使用 Redis 时，三台机器共享同一个缓存实例，更新一次，全局可见。

| 对比项 | 本地缓存（Local Cache） | Redis 缓存（Distributed Cache） |
| --- | --- | --- |
| **存储位置** | 应用进程内存中（JVM 内） | 独立 Redis 服务（进程外） |
| **访问速度** | 纳秒级（无网络开销） | 微秒级（需要网络 IO） |
| **数据范围** | 单机有效，不共享 | 分布式共享，多节点可访问 |
| **一致性** | 各节点之间数据可能不一致 | 数据集中存储，一致性较好 |
| **容量大小** | 受应用内存限制 | 可独立扩容，容量更大 |
| **持久化** | 一般不支持 | 支持 RDB/AOF 持久化 |
| **适用场景** | 高频、本地热点数据 | 跨服务共享数据，全局缓存 |


本地缓存：

是指将**数据存储在本地应用程序或服务器上，通常用于加速数据访问和提高响应速度。**

本地缓存通常**使用内存作为存储介质，利用内存的高速读写特性来提高数据访问速度。**

本地缓存的**优势**：

**访问速度快**：由于本地缓存存储在本地内存中，因此访问速度非常快，能够满足频繁访问和即时响应的需求。

**减轻网络压力**：本地缓存能够降低对远程服务器的访问次数，从而减轻网络压力，提高系统的可用性和稳定性。

**低延迟**：由于本地缓存位于本地设备上，因此能够提供低延迟的访问速度，适用于对实时性要求较高的应用场景。

本地缓存的**不足**：

可扩展性有限：本地缓存的可扩展性受到硬件资源的限制，无法支持大规模的数据存储和访问。

**分布式缓存（Redis）**：

是指**将数据存储在多个分布式节点上，通过协同工作来提供高性能的数据访问服务。**

分布式缓存通常使用集群方式进行部署，利用多台服务器来分担数据存储和访问的压力。

分布式缓存的**优势**：

**可扩展性强**：分布式缓存的节点可以动态扩展，能够支持大规模的数据存储和访问需求。

**数据一致性高**：通过分布式一致性协议，分布式缓存能够保证数据在多个节点之间的一致性，减少数据不一致的问题。

**易于维护**：分布式缓存通常采用自动化管理方式，能够降低维护成本和管理的复杂性。

分布式缓存的**不足**：

**访问速度相对较慢**：相对于本地缓存，分布式缓存的访问速度相对较慢，因为数据需要从多个节点进行访问和协同。

**网络开销大**：由于分布式缓存需要通过网络进行数据传输和协同操作，因此相对于本地缓存来说，网络开销较大。

### 那什么时候用本地缓存，什么时候用 Redis 呢？
+ **本地缓存适合：** 高频、热点、小数据量、读多写少、对一致性要求不高的场景（如配置、用户 Token 等）。
+ **Redis 缓存适合：** 多节点共享、需要持久化或高一致性的场景（如分布式系统共享 Session、排行榜、库存数据等）。

实际生产中常用 **两级缓存（Local + Redis）** 结合方式：

+ 先读本地缓存（命中则返回）；
+ 没命中再查 Redis；
+ Redis 更新时通过 **消息通知或失效策略** 同步本地缓存。

### 高并发场景，Redis单节点+MySQL单节点能有多大的并发量？
具体数值要看压测环境，但总体上 Redis 的并发能力比 MySQL 高几个数量级。  
我会通过缓存、分库分表、异步化等手段让系统整体能支撑更高的并发。

### redis应用场景是什么？
Redis 是一种基于内存的数据库，对数据的读写操作都是在内存中完成，因此读写速度非常快，常用于**缓存，消息队列、分布式锁**等场景。

+ 缓存: Redis最常见的用途就是作为缓存系统。通过将热门数据存储在内存中，可以极大地提高访问速度，减轻数据库负载，这对于需要快速响应时间的应用程序非常重要。
+ 排行榜: Redis的有序集合结构非常适合用于实现排行榜和排名系统，可以方便地进行数据排序和排名。
+ 分布式锁: Redis的特性可以用来实现分布式锁，确保多个进程或服务之间的数据操作的原子性和一致性。
+ 计数器 由于Redis的原子操作和高性能，它非常适合用于实现计数器和统计数据的存储，如网站访问量统计、点赞数统计等。
+ 消息队列: Redis的发布订阅功能使其成为一个轻量级的消息队列，它可以用来实现发布和订阅模式，以便实时处理消息。

### Redis支持并发操作吗？
**单个命令的原子性**：  
 每条命令在执行时是不可分割的，要么全部执行成功，要么不执行，从而避免并发冲突。

**多个操作的原子性**：  
 如果需要让多条命令具备原子性，可以使用 **事务（MULTI / EXEC / WATCH / DISCARD）** 或 **Lua 脚本**，  
 Redis 会**把整个事务或脚本当作一个整体来执行**，执行过程中不会被其他命令打断，从而保证原子性。

不过需要注意的是，Redis 的事务并不像传统数据库那样支持回滚（rollback），  
一旦执行 `EXEC`，**即使中途有命令失败，其他命令也会继续执行。**

所以在实际项目中，**如果要严格保证多步操作的原子性**，一般会使用 **Lua 脚本** 来实现。

### Redis分布式锁的实现原理？什么场景下用到分布式锁？
 就是在微服务集群里，**为了防止多台机器同时去干同一件事情，导致数据错乱或者资源浪费**。  

**防止并发扣减引发的超卖：** 比如电商场景下的库存扣减，或者用户账户的余额变更。多个并发请求打过来，如果不加锁，读写就会产生脏数据。因为是多个微服务节点，Go 原生的 `sync.Mutex` 只能锁住当前进程，跨机器就失效了，必须依赖 Redis 这个公共中间件来做协调。  

**防止定时任务重复执行：** 假设我们有个服务要每分钟去拉取最新的汇率数据。为了高可用，这个服务部署了 3 个 Pod（实例）。时间一到，3 个实例同时触发拉取逻辑，这不仅浪费性能，还可能向数据库重复插数据。这时候谁先抢到 Redis 里的那把锁，谁就去干活，没抢到的直接 return。



Redis 分布式锁主要通过 `SET key value NX EX time` 实现，  
利用 Redis 的单线程特性保证命令原子性。  
当多台机器竞争同一个资源时，**谁能成功设置这个 key，谁就拿到锁。**

常用于分布式系统中防止重复操作，比如秒杀、库存扣减、任务调度等场景。

Redis 分布式锁通常有两种实现方式：

1.基于 `SET NX EX` 的简单实现

获取锁：

```go
SET lock_key unique_value NX EX 10
```

`NX`：key 不存在时才设置，保证只有一个客户端能拿到锁；

`EX 10`：设置过期时间，防止死锁；

`unique_value`：通常是请求唯一标识（如 UUID），用于解锁时校验是否自己持有。

释放锁（保证原子性）：

```plain
// 释放锁时，先比较 unique_value 是否相等，避免锁的误释放
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

使用 **Lua 脚本** 确保“判断 + 删除”是原子操作，防止误删他人锁。

2. Redlock 算法
+ 由 Redis 作者提出的更安全实现方案；
+ 在多个 Redis 节点上同时加锁（如 5 个节点至少 3 个成功）才认为加锁成功；
+ 用于提高在 Redis 集群中锁的可靠性。

Redis 分布式锁常用于多节点、多线程共享资源的场景：

1. **秒杀 / 抢购活动**：**防止库存被超卖。**
2. **定时任务 / 分布式调度**：**确保同一任务只被一个节点执行**。
3. **防重复提交**：**防止用户多次点击重复下单。**
4. **并发更新资源**：**如用户账户余额修改、商品库存更新等。**

### 用go语言实现一下redis的分布式锁
我会使用 Redis 的 `SETNX` 命令来实现分布式锁。  
 加锁时，用 `SET key value NX PX expireTime` 保证原子性：

+ `NX` 表示 key 不存在时才设置，避免重复加锁；
+ `PX` 设置过期时间，防止死锁；
+ `value` 设置为唯一标识（如 UUID），用于解锁时验证锁归属。

解锁时，我会用 **Lua 脚本** 来保证原子性地判断和删除锁：

```go
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

这样可以防止误删他人的锁。

整个过程保证了：

1. 加锁原子性（通过 `SETNX`）；
2. 解锁原子性（通过 Lua 脚本）；
3. 互斥性（同一时刻只有一个客户端持锁）；
4. 防死锁（自动过期机制）；
5. 正确解锁（通过唯一 value 区分不同客户端）。

### Redis的大Key问题是什么？
Redis 的大 Key 问题是指**单个 Key 存储的数据量过大**，比如**一个 Key 对应很大的字符串或非常多的元素。**

大 Key 会导致操作这个 Key 时阻塞 Redis 主线程，从而影响其他请求的响应，严重时会导致 Redis 卡顿甚至主从复制延迟。

### 大Key问题的缺点？
大 Key 的主要缺点是会造成 **性能阻塞**。  
因为 Redis 是单线程的，操作大 Key（比如读取、删除、序列化）会占用主线程较长时间，导致其他命令延迟。  
此外还可能引发 **内存不均、网络带宽压力、主从复制延迟** 等问题。

+ **内存占用过高**。大Key占用过多的内存空间，可能导致可用内存不足，从而触发内存淘汰策略。在极端情况下，可能导致内存耗尽，Redis实例崩溃，影响系统的稳定性。
+ 性能下降。大Key会占用大量内存空间，导致内存碎片增加，进而影响Redis的性能。对于大Key的操作，如读取、写入、删除等，都会消耗更多的CPU时间和内存资源，进一步降低系统性能。
+ 阻塞其他操作。某些对大Key的操作可能会导致Redis实例阻塞。例如，使用DEL命令删除一个大Key时，可能会导致Redis实例在一段时间内无法响应其他客户端请求，从而影响系统的响应时间和吞吐量。
+ 网络拥塞。每次获取大key产生的网络流量较大，可能造成机器或局域网的带宽被打满，同时波及其他服务。例如：一个大key占用空间是1MB，每秒访问1000次，就有1000MB的流量。
+ 主从同步延迟。当Redis实例配置了主从同步时，大Key可能导致主从同步延迟。由于大Key占用较多内存，同步过程中需要传输大量数据，这会导致主从之间的网络传输延迟增加，进而影响数据一致性。
+ 数据倾斜。在Redis集群模式中，某个数据分片的内存使用率远超其他数据分片，无法使数据分片的内存资源达到均衡。另外也可能造成Redis内存达到maxmemory参数定义的上限导致重要的key被逐出，甚至引发内存溢出。

### 大Key的检测方式
**第一招：利用原生工具快速摸底 (**`**redis-cli --bigkeys**`**)** 如果是想快速看一下集群的大致情况，我会直接使用 Redis 自带的 `--bigkeys` 参数。

+ **做法：** 直接在命令行运行 `redis-cli -h <ip> -p <port> --bigkeys`。
+ **原理与优点：** 它内部使用的是 `SCAN` 命令，而不是会卡死主线程的 `KEYS *`，所以对线上的阻塞影响比较小。它会按数据类型分别给你列出一个 Top 1 的大 Key。
+ **避坑（重点）：** 它的缺点也很明显，第一，它只能找出每种数据类型里**最大的那一个**，没法帮你找出所有的 BigKey；第二，对于集合类型，它统计的是‘元素个数’而不是真实的‘内存占用’，有时候元素多不代表真正占用内存大。

**第二招：离线 RDB 快照分析（大厂标准的终极杀器）**

这是我们在生产环境中最安全、也最彻底的排查方式，主打一个“对线上业务零干扰”。

+ **做法：** 首先在流量低谷期，对线上的 Redis 触发一次 `bgsave`，生成一份全量的 RDB 备份文件。然后把这个 RDB 文件拉到一台没有业务流量的线下机器上，使用业界成熟的开源工具——比如 Python 写的 `**redis-rdb-tools**`（或者 Go 写的 `rdb` 解析库）来进行离线解析。
+ **优点：** 这个工具可以直接把 RDB 里的数据解析并输出成一份 CSV 报表。在这份报表里，所有的 Key、数据类型、占用字节数（Bytes）、元素个数一目了然。我们甚至可以直接按字节数做个降序排列，全盘掌握所有的 BigKey，精准定位。

**第三招：脚本扫描或云厂商自带监控（自动化防御）** 如果在日常维护中：

+ 如果只有少数几个特定的 Key 存在嫌疑，我会在业务代码或脚本中，配合 `SCAN` 命令分批迭代，再分别调用 `MEMORY USAGE key` （查具体字节数）或者 `HLEN/ZCARD` 来单独排查。
+ 如果使用的是公有云（比如阿里云、腾讯云等），它们的 Redis 控制台通常都自带了“大 Key 分析”的可视化监控大屏，底层其实也是基于全量 RDB 或者节点分析做的，有现成工具可以直接拿来用。

**总结一下：** 快速摸底用 `--bigkeys`，线上彻底排查用 **RDB 离线分析**，这样既能拿到精准数据，又不会把线上的主线程拖垮。

### Redis大key如何解决？
Redis 大 Key 的主要解决思路是：

**避免产生大 Key、分拆大 Key、异步删除大 Key**。

+ **对大Key进行拆分**。例如将含有数万成员的一个**HASH Key拆分为多个HASH Key**，并确保每个Key的成员数量在合理范围。在Redis集群架构中，拆分大Key能对数据分片间的内存平衡起到显著作用。
+ **对大Key进行清理**。将不适用Redis能力的数据存至其它存储，并在Redis中删除此类数据。注意，要使用异步删除。
+ **监控Redis的内存水位**。可以通过监控系统设置合理的Redis内存报警阈值进行提醒，例如Redis内存使用率超过70%、Redis的内存在1小时内增长率超过20%等。
+ **对过期数据进行定期清理**。堆积大量过期数据会造成大Key的产生，例如在HASH数据类型中以增量的形式不断写入大量数据而忽略了数据的时效性。可以通过定时任务的方式对失效数据进行清理。



### 什么是热key？
通常以其**接收到的Key被请求频率来判定**

热 Key 指的是在 Redis 中**被高频访问的 Key，**  
某个 Key 的访问量远高于其他 Key，造成访问集中，  
可能导致 Redis 实例或**节点负载过高，出现延迟或热点倾斜问题。**

> 比如一个秒杀商品库存 Key、热门榜单数据 Key、活动配置 Key 等。
>

| 问题类型 | 说明 |
| --- | --- |
| **1. 请求集中，导致单节点压力过大** | Redis 是单线程的，某个热 Key 集中在一个节点上，会造成该节点 QPS 过高，响应变慢甚至宕机。 |
| **2. 集群数据倾斜** | 在 Redis Cluster 中，如果某个槽上存放了热 Key，该节点会成为性能瓶颈。 |
| **3. 缓存击穿风险** | 热 Key 过期后，瞬间大量请求同时访问数据库，造成缓存击穿甚至 DB 崩溃。 |


### 如何解决热key问题？
比如在汇率资讯系统中，如果某个重磅金融新闻发布，瞬间有几百万用户同时下拉刷新拉取同一篇文章，这个 `article:info:10086` 就会成为极端的读热 Key。

**第一部分：应对“读热点”（最常见的场景）**

+ **大招 1：引入本地缓存（Local Cache）+ Singleflight 兜底** 

这是对抗读热 Key 最有效、最彻底的手段。

我会直接在 Go 业务进程里引入本地缓存（比如 `bigcache` 或 `freecache`）。当请求打到微服务网关时，优先查 Go 进程的本地内存。只要本地有，直接返回，彻底斩断对 Redis 的流量。 

为了防止本地缓存失效的瞬间，海量请求同时去查 Redis（引发缓存击穿），我一定会配合 Go 的 `golang.org/x/sync/singleflight` 使用。

把对同一个热 Key 的请求合并，保证同一时刻**只有一个 Goroutine** 真正发起对 Redis 的网络调用，拿到数据后共享给其他阻塞等待的协程。在 Go-zero 等框架中，这种机制非常容易通过中间件或拦截器落地 。

+ **大招 2：热 Key 打散（Sharding 复制）** 

如果业务要求强一致性，本地缓存延迟无法接受，我会采用“打散复制”的方案。

把原本的一个热 Key 复制多份，比如变成 `article:10086:node1` 到 `article:10086:node10`。

这 10 个副本利用 Hash 槽的特性会分散在 Redis Cluster 的不同物理节点上。

前端请求打过来时，业务层在 Go 代码里生成一个 1 到 10 的随机数，随机去读其中一个副本。这样就把单台机器的 QPS 压力完美均摊到了 10 台机器上。

**第二部分：应对“写热点”** 写热点比较少见但更棘手，比如某篇爆款文章瞬间涌入几百万次点赞，大家都往同一个 ZSet 或同一个计数器里疯狂写入。

+ **大招 1：本地合并写（批量 Flush）** 

利用 Go 语言高并发的优势，我会开启一个带 Buffer 的 Channel 和一个后台 Goroutine。

用户的点赞请求不直接打给 Redis，而是全部扔进 Channel。后台 Goroutine 每隔 100 毫秒或者攒够 1000 个赞，把这 1000 个 +1 聚合成一次 `INCRBY key 1000` 发给 Redis。网络 I/O 次数瞬间降低三个数量级。

+ **大招 2：分片写入 + 读合并** 
+ 类似于处理大数据的 MapReduce 思想，我会把一个计数器拆成多个。比如把 `article_likes` 拆成 `article_likes:01` 到 `article_likes:10`。用户的点赞请求随机写入这 10 个分片中的一个。当需要读取总赞数时，用 `MGET` 把 10 个分片的值同时拉回来，在 Go 业务层做一次简单的 `Sum` 聚合运算。

**总结：** 解决热 Key 的核心思想就两个字：**拦截**（用本地缓存挡住）和**分治**（用打散和合并来均摊压力）。



在 Redis 集群中，热 Key 的访问集中在单个分片上，因为 Redis 的哈希槽机制导致一个 Key 只能属于一个节点。  
为了解决这种单分片热点问题，可以采用两种方式：

**1️⃣**** 复制热 Key 到多个节点（数据副本法）**  
 把一个热 Key（如 `foo`）复制为 `foo2`、`foo3` 等副本，分布到不同分片。  
 客户端读取时随机访问这些副本，实现读流量分摊。  
 这种方式可以显著降低单节点压力，但需要业务层逻辑支持。

**2️⃣**** 使用读写分离架构**  
 如果热 Key 是由**读操作过多**引起的，可以为该分片增加多个从节点，  
 让读请求走从节点，写请求仍走主节点。  
 不过这种方式会增加架构复杂度，比如要用代理层（LVS、Proxy）做读流量负载，  
 同时要考虑从节点增多带来的**延迟同步与维护成本**。

### <font style="color:#DF2A3F;background-color:#FBDE28;">如何保证 redis 和 mysql 数据缓存一致性问题？</font>
在实际落地时，我通常会从基础策略、异常兜底和极端并发控制三个层面来设计：

**第一层：基础策略（Cache Aside Pattern 旁路缓存）** 这是业界的标准基线。

+ **读请求：** 先查 Redis，命中直接返回；没命中就去查 MySQL，然后把数据回写到 Redis，设置个合理的过期时间。
+ **写请求（核心）：** 必须是**先更新 MySQL，再删除 Redis**。
+ **（主动防御面试官的追问：为什么是删除而不是更新？）** 因为如果改成‘更新’缓存，在并发写的情况下极其容易发生线程执行顺序错乱，导致脏数据永久驻留。而且很多复杂的业务（比如连表计算出来的结果），每次修改都去算一遍更新缓存，性能极差。‘删除’本质上是一种‘懒加载’思想，谁下一次查，谁再去算。

**第二层：异常兜底（解决删缓存失败的问题）** 哪怕我们规定了先更新 DB 再删缓存，如果第二步**网络闪断、删缓存失败了怎么办？**此时 DB 是新数据，Redis 是老数据，一致性就被破坏了。 我会采用两种方案来兜底：

+ **方案 A：基于消息队列（MQ）的异步重试。** **业务代码更新完 MySQL 后，如果删 Redis 失败，就把这个 Key 扔进消息队列（比如 RabbitMQ）。消费者端不断消费去重试删除，直到成功。优点是实现简单，缺点是代码有侵入性。**
+ **方案 B：订阅 Binlog 异步删除（业界最优雅的解法）。** 这是我个人在复杂系统中更倾向的做法。业务代码只管写 MySQL，完全不管缓存。我们在底层额外部署一个组件（比如 Canal），把自己伪装成 MySQL 的从节点，监听 Binlog 的变更日志。一旦发现数据有改动，立刻提取出对应的 Key 去异步清理 Redis 。这种方案与主业务代码 100% 解耦。

**第三层：极端并发控制（解决经典的读写冲突）** 即使删缓存没有失败，在超高并发下依然有个致命的极端场景： 线程 A 准备更新数据；线程 B 刚好来查数据。此时 A 还没来得及删缓存，B 查缓存没命中，去了数据库查到了**旧数据**，正准备往 Redis 里写；紧接着，A 删除了缓存，然后 B 把那份**旧数据**堂而皇之地写进了 Redis。脏数据再次诞生。

+ **高阶解法：延迟双删（Delayed Double Delete）。** 为了防住这种极端并发，我会采用延迟双删策略。伪代码逻辑是：
    1. 先删除一次缓存。
    2. 更新 MySQL。
    3. 休眠一小段时间（比如 500 毫秒，具体时间取决于读请求耗时）。
    4. **再删除一次缓存。** 这第二次删除，就是为了把休眠期间其他线程写进去的旧脏数据给干掉。当然，这第二次删除也可以结合前面的 MQ 或 Binlog 异步去做，避免休眠阻塞当前业务线程。



对于 **读操作**，我采用的是 **Cache Aside（旁路缓存）策略**：  
 **应用程序先从缓存中读取数据，如果缓存未命中，再从数据库加载数据并写回缓存。**  
 这种方式能有效利用缓存命中率来提升读取性能。

对于 **写操作**，我选择 **先更新数据库，再删除缓存** 的方案。  
 **这样可以保证当缓存和数据库数据不一致时，下一次读取请求会从数据库重新加载最新数据。**

缓存系统本质上是通过 **牺牲强一致性来换取高性能和高可用性**，  
 根据 **CAP 理论**，Redis 属于 **AP 模型**（可用性 + 分区容忍性），  
 因此它更适用于**对一致性要求不高的场景**。  
 如果业务必须保持**强一致性**，那其实就不应该使用缓存层

**为了防止在“删除缓存失败”或“网络异常”时出现数据不一致，**  
 我会使用以下两种补偿方案：

1️⃣ **删除缓存重试机制（消息队列）**  
 数据库更新成功后，将“删除缓存”的操作异步投递到消息队列（如 Kafka、RabbitMQ），  
 由消费者重试执行缓存删除，保证最终删除成功

2️⃣ **订阅 Binlog 异步删除缓存（Canal + MQ）**  
 通过 Canal 订阅 MySQL 的 Binlog 日志，当数据库表数据发生变化时，  
 由 Canal 将变更消息发送到消息队列，再由消费者根据表名和主键删除对应缓存。  
 这种方式对业务代码无侵入，常用于高并发环境下的缓存同步。

### 为什么会出现不一致？
因为 Redis 是**缓存层**，MySQL 是**持久层**，  
 在 **并发写入** 或 **缓存更新失败** 的情况下，可能出现：

+ 缓存中是旧值；
+ 数据库中是新值；
+ 或者缓存比数据库先更新。

### 缓存雪崩、击穿、穿透是什么？怎么解决？
+ 缓存雪崩：大量Key过期

**当大量缓存数据在同一时间过期（失效）或者 Redis 故障宕机时**，如果此时**有大量的用户请求，都无法在 Redis 中处理，于是全部请求都直接访问数据库，从而导致数据库的压力骤增，严重的会造成数据库宕机，从而形成一系列连锁反应，造成整个系统崩溃**，这就是缓存雪崩的问题。

**缓存雪崩解决方案：**

**均匀设置过期时间**：如果要给缓存数据设置过期时间，应该避免将大量的数据设置成同一个过期时间。我们可以在对缓存数据设置过期时间时，给这些数据的过期时间加上一个随机数，这样就保证数据不会在同一时间过期。

**互斥锁**：当业务线程在处理用户请求时，如果发现访问的数据不在 Redis 里，就加个互斥锁，保证同一时间内只有一个请求来构建缓存（从数据库读取数据，再将数据更新到 Redis 里），当缓存构建完成后，再释放锁。未能获取互斥锁的请求，要么等待锁释放后重新读取缓存，要么就返回空值或者默认值。实现互斥锁的时候，最好设置超时时间，不然第一个请求拿到了锁，然后这个请求发生了某种意外而一直阻塞，一直不释放锁，这时其他请求也一直拿不到锁，整个系统就会出现无响应的现象。

**后台更新缓存**：业务线程不再负责更新缓存，缓存也不设置有效期，而是让缓存“永久有效”，并将更新缓存的工作交由后台线程定时更新。

**① 给 Key 过期时间加随机值；② 设置热点数据永不过期；③ 多级缓存（本地 + Redis）；④ 限流、熔断、降级**

****

+ **缓存击穿：热点Key失效过期**

**<font style="background-color:#FBDE28;">如果缓存中的某个热点数据过期了，此时大量的请求访问了该热点数据，就无法从缓存中读取，直接访问数据库，数据库很容易就被高并发的请求冲垮</font>****，这就是缓存击穿的问题。**

**缓存击穿解决方案：**

**互斥锁方案**，保证同一时间只有一个业务线程更新缓存，未能获取互斥锁的请求，要么等待锁释放后重新读取缓存，要么就返回空值或者默认值。

**不给热点数据设置过期时间**，**由后台异步更新缓存，或者在热点数据准备要过期前，提前通知后台线程更新缓存以及重新设置过期时间；**

**① 热点 Key 永不过期；② 使用分布式锁或互斥锁防止并发加载；③ 预热缓存**

****

+ **缓存穿透：数据即不在缓存也不在数据库**

<font style="background-color:#FBDE28;">当用户访问的数据，</font>**<font style="background-color:#FBDE28;">既不在缓存中，也不在数据库中</font>**<font style="background-color:#FBDE28;">，</font>**<font style="background-color:#FBDE28;">导致请求在访问缓存时，发现缓存缺失，再去访问数据库时，发现数据库中也没有要访问的数据，没办法构建缓存数据，来服务后续的请求</font>****。那么当有大量这样的请求到来时，数据库的压力骤增，这就是缓存穿透的问题。**

**缓存穿透解决方案：**

**非法请求的限制**：当有大量恶意请求访问不存在的数据的时候，也会发生缓存穿透，因此在 API 入口处我们要判断求请求参数是否合理，请求参数是否含有非法值、请求字段是否存在，如果判断出是恶意请求就直接返回错误，避免进一步访问缓存和数据库。

**缓存空值或者默认值**：当我们线上业务发现缓存穿透的现象时，可以针对查询的数据，**在缓存中设置一个空值或者默认值，这样后续请求就可以从缓存中读取到空值或者默认值，返回给应用，而不会继续查询数据库**。

**布隆过滤器**：我们可以在写入数据库数据时，**使用布隆过滤器做个标记**，然后在用户请求到来时，业务线程确认缓存失效后，可以通过**查询布隆过滤器快速判断数据是否存在**，如果不存在，就不用通过查询数据库来判断数据是否存在。即使发生了缓存穿透，大量请求只会查询 Redis 和布隆过滤器，而不会查询数据库，保证了数据库能正常运行，Redis 自身也是支持布隆过滤器的

**① 使用布隆过滤器过滤无效 Key；② 缓存空值；③ 参数合法性校验；④ 接口限流**

**总结：**

雪崩、击穿、穿透的本质分别是：**批量失效、单点失效、无效请求**。  
对应的解决思路是：**错峰过期 + 锁机制 + 预过滤**。  
重点在于削峰填谷、提前预防、快速恢复。

### 布隆过滤器原理介绍一下
**第一：布隆过滤器的核心定律（一句话概括）** 

它可以极其高效地判断一个元素是否在一个集合中。

它的核心定律只有两句话： 

**‘它说某个数据不存在，那这个数据一定不存在。’**

**‘它说某个数据存在，那这个数据可能存在（有一定误判率）。’**

**第二：底层工作原理（它是怎么做到的？）** 

布隆过滤器的底层其实就是一个巨大的 **Bitmap（位图）**，外加**多个不同的 Hash 函数** 。

+ **写入（Add）过程：** 假设我们要把用户 ID `10086` 存进去。我们会让这一个 ID 经过 3 个不同的 Hash 函数计算，得出 3 个不同的下标位置（比如 2、7、14），然后把 Bitmap 上这 3 个位置的 bit 全部置为 `1`。
+ **查询（Check）过程：** 当我们要查 `10086` 在不在时，同样用这 3 个 Hash 函数算一遍，看看下标 2、7、14 对应的值是不是**全部都是 **`**1**`。如果全是 `1`，说明它**大概率**在。
+ **为什么会有误判？（极佳加分项）：** 假设我们查另一个不存在的 ID `10010`，它算出来的 Hash 结果刚好也是 2、7、14。虽然我们从来没把 `10010` 存进去过，但因为之前存 `10086` 等其他数据时，把这些位置都置为 `1` 了（Hash 碰撞/位重叠），系统就会误报它存在。这就是布隆过滤器‘存在误判’的根本原因。

**第三：工程实战与致命缺陷（结合 Go-Zero 经验）**

在实际工程中，布隆过滤器最经典的应用场景就是**解决缓存穿透**。 

比如在我用 Go-Zero 开发微服务网关和 RPC 接口时 ，如果有黑客伪造大量数据库里根本不存在的 `article_id` 疯狂发起请求。

这些请求会穿透 Redis，直接把底层的 MySQL 打挂 。 

这时候，我会在 Redis 前面或者 Go 的业务层挡一个布隆过滤器。

所有存在的 `article_id` 都在里面。

黑客的请求过来，布隆过滤器一查，发现这几个 Hash 位不全是 `1`，立刻判定**‘绝对不存在’**，直接在网关层丢弃请求，连 Redis 都不用去访问，极其高效地保护了底层存储。

**（主动抛出缺陷）：但在工程落地时，它有一个致命的坑：原生的布隆过滤器不支持删除（Delete）。** 

因为多个元素的 Hash 结果可能会映射到同一个 bit 位上。

如果我把某篇文章下架了，想在布隆过滤器里把它的 bit 位改成 `0`，很可能会导致其他正常文章的查询也跟着失效。为了解决这个问题，如果业务确实需要频繁删除，我会考虑引入 **Counting Bloom Filter（计数布隆过滤器）**，或者定期通过跑离线批处理任务，重建一个全新的布隆过滤器来替换旧的。



布隆过滤器是一种**基于位数组和多哈希函数的概率型数据结构**，  
 用来判断一个元素是否“可能存在”或“一定不存在”。  
 它能在很小的内存下高效过滤无效请求，但有一定的**误判率**。  
 在 Redis 中常用于防止**缓存穿透**。

布隆过滤器由「初始值都为 0 的位图数组」**和**「 N 个哈希函数」两部分组成。

当我们在写入数据库数据时，在布隆过滤器里做个标记，这样下次查询数据是否在数据库时，只需要查询布隆过滤器，如果查询到数据没有被标记，说明不在数据库中。

### 如何设计秒杀场景处理高并发以及超卖现象？
秒杀系统要同时解决两个核心问题：

1. **高并发问题** —— 大量用户瞬时请求，防止系统崩溃；
2. **超卖问题** —— 防止库存出现负数或卖出超过库存量。

#### 核心架构设计思路
### 1️⃣ 前端层限流与削峰
+ **前端限流**：在页面或接口层做限流，比如验证码、按钮防重复提交、Token 校验。
+ **Nginx 限流**：使用 Nginx 的限速模块（limit_req），控制每秒进入系统的请求数量。
+ **消息队列削峰**：将请求写入 MQ（如 Kafka / RabbitMQ），由后台异步处理，防止瞬时打爆 Redis 或数据库。

### 2️⃣ Redis 缓存层 —— 高速扣减库存
**关键设计**：库存预热 + 原子扣减

+ 将商品库存提前加载到 Redis；
+ 扣减时执行原子操作：

```plain
-- Lua脚本保证原子性
local stock = redis.call("GET", KEYS[1])
if (tonumber(stock) <= 0) then
    return 0
else
    redis.call("DECR", KEYS[1])
    return 1
end
```

+ Redis 是单线程的，执行 Lua 脚本能保证原子扣减，不会出现并发超卖。

### 3️⃣ 异步落库（防止DB被打爆）
+ Redis 成功扣减后，发送消息到 MQ；
+ 消费者异步执行数据库扣库存操作；
+ 最终数据库与 Redis 数据保持一致（可通过延迟任务或定时校准）。

| 策略 | 说明 |
| --- | --- |
| ✅ **Redis Lua 脚本原子扣减** | 保证并发下库存只会被正确减少一次。 |
| ✅ **分布式锁（Redisson）** | 适用于需要加锁的复杂业务逻辑场景。 |
| ✅ **库存预热 + 预减库存** | 活动前将库存同步至 Redis，在请求阶段只访问缓存。 |
| ✅ **消息队列异步落库** | 防止数据库瞬时压力过大。 |
| ✅ **异步回补** | Redis 中库存扣减成功但下单失败的场景要回补库存。 |


| 追问 | 答法要点 |
| --- | --- |
| ❓ Redis 宕机怎么办？ | 使用 Redis 主从 + 哨兵，保证高可用；关键库存可持久化（RDB/AOF）。 |
| ❓ 如何防止重复下单？ | 使用用户ID + 商品ID 建唯一Key，SETNX实现幂等。 |
| ❓ 数据一致性怎么保证？ | Redis 扣减成功 → MQ → MySQL 更新库存 → 定时校验数据一致性。 |
| ❓ 秒杀结束如何清理缓存？ | 活动结束后统一删除 Redis 库存Key或标记过期。 |


在秒杀场景下，为了应对高并发和防止超卖，我会：

+ 使用 **Nginx + 消息队列削峰限流**；
+ 将商品库存 **预热到 Redis**；
+ 使用 **Lua 脚本原子扣减库存** 保证并发安全；
+ 成功扣减后通过 **MQ 异步写入数据库**；
+ 并在最后做 **库存一致性校验与回补**。  
   这样能保证系统高性能且避免超卖。
